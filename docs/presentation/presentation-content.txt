Lessons learned:

Enterprise Anpassungen BMAD
* Reduktion des Kontext als allg. Problem
* DDD Ansatz
* es scheint ein schritt zu fehlen. nach pro, werden direkt epics und stories definiert. es fehlen aber pro domain 
* aufteilung in Domains und weiterarbeiten nur in einer Domain oder sogar subdomain und doch Konsistenz über ganze Applikation behalten.
* APIs pro domain. api konsolidieren und so definieren, dass grosser reuse durch frontend wireframes
* Wireframes mit den benötigten APIs für lesen schreiben und actions
* Wireframes pro screen mit der Navigation zu anderen screens
* TDD Red-Green-Refactor. No Mocks on src 
* Infrastruktur Stories am Anfang
* aws stacks korrekt aufsetzen. Zertifikate, CNAME records, alles jeweils im richtigen account.
* accounts pro environment
* DEV Test Prod Automatisierung
* auch services deployen 
* Storyboard der screenzusammenhänge
* Stories mit Screens, APIs, datenmodell wie Detailspec
* Start with small mvp aller domains (heathcheck)
* DONT use story numbers with meaning. this creates cofusion when refactoring or reordering stories. use unspecific up counting ID’s like Jira does
* update template to have clear handling of data entities in DB, JPA and Backend.
* use opendoc types und lass backend und frontend dagegen entwickeln
* klare Aufteilung in unit, integration, e2e tests
