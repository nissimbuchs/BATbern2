## Review Summary

Story 6.2b is well-specified and correctly aligned with ADR-004 (Factor User Fields from Domain Entities). The cross-service sync pattern is properly documented, with User fields (firstName, lastName, bio, photo) routed to Company Service and Speaker fields (expertise, topics, languages, linkedInUrl) stored locally in Event Service.

## Findings

### Strengths

1. **ADR-004 Compliance**: Correctly implements User/Speaker field separation pattern
2. **ADR-002 Compliance**: Photo upload uses presigned URL pattern
3. **Comprehensive API Spec**: 4 endpoints with full request/response schemas
4. **Clear Field Ownership**:
   - User entity (Company Service): firstName, lastName, bio, profilePictureUrl
   - Speaker entity (Event Service): expertiseAreas, speakingTopics, languages, linkedInUrl
5. **Profile Completeness Formula**: Clear weightings (15+15+20+20+15+15 = 100%)
6. **30 Pre-defined Tests**: 15 backend + 15 frontend

### Issues Identified

1. **Token Type Ambiguity** (Medium Impact)
   - Story uses "magic link token" without specifying type
   - Story 6.1a defined RESPOND and PROFILE token types
   - Question: Can RESPOND token be reused for profile access after acceptance?
   - Recommendation: Clarify token requirements before implementation

2. **Access Control Gap** (Low Impact)
   - Story lists 6.2a as dependency (speaker must be ACCEPTED)
   - This isn't enforced in API spec or acceptance criteria
   - Recommendation: Document whether INVITED speakers can access profile

3. **Partial Sync Failure Strategy** (Medium Impact)
   - AC10.4 states "Partial save allowed if one service fails (with warning)"
   - Rollback strategy unclear:
     - If Speaker fields save but Company Service fails, what happens?
     - Is this transaction rollback, compensation, or accept partial success?
   - Recommendation: Document rollback/compensation strategy

### Test Gaps

| No concurrent update test | Low | Test for race condition handling |

## Recommendations

### Before Implementation (Required)

1. **Clarify token requirements**: Document which token type(s) grant profile access

### During Implementation (Should Do)

2. **Document sync failure strategy**: Choose between:
   - Transaction rollback (all-or-nothing)
   - Partial success with warning (current spec)
   - Retry with exponential backoff
3. **Add sync failure integration test**: Test Company Service unavailable scenario
4. **Document @Transactional boundaries**: Clarify what's atomic

### Nice to Have

5. **Clarify access control**: Can INVITED speakers access profile?
6. **Add performance SLA**: Target profile save < 1s including sync
