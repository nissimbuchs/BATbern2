# Story 5.x+1: RAG-Powered Materials Search with AWS Bedrock

Accepted

## Story
**As a** Public user browsing the archive,
**I want** to search through all session materials content using natural language queries,
**so that** I can quickly find relevant presentations, documents, and discussions across all archived events without manually downloading each file.

**Proposed Schema Extensions:**

```sql
-- Enable pgvector extension (if not already enabled)
CREATE EXTENSION IF NOT EXISTS vector;

-- Store extracted text chunks for semantic search
CREATE TABLE material_content_chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    material_id UUID NOT NULL REFERENCES session_materials(id) ON DELETE CASCADE,
    chunk_index INTEGER NOT NULL,                -- Sequential chunk number within material
    content TEXT NOT NULL,                       -- Extracted text chunk (500 tokens with 50 overlap)
    page_number INTEGER,                         -- For PDFs (which page this chunk is from)
    slide_number INTEGER,                        -- For presentations (which slide)
    timestamp_seconds INTEGER,                   -- For videos (timestamp in seconds)
    token_count INTEGER NOT NULL,                -- Number of tokens in this chunk
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(material_id, chunk_index),
    INDEX idx_material_chunks_material_id (material_id)
);

-- Store vector embeddings for each chunk
CREATE TABLE material_embeddings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    chunk_id UUID NOT NULL REFERENCES material_content_chunks(id) ON DELETE CASCADE,
    embedding vector(1024),                      -- Bedrock Titan Embeddings G1 (1024 dimensions)
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_material_embeddings_chunk_id (chunk_id)
);

-- Create HNSW index for fast vector similarity search (cosine distance)
CREATE INDEX idx_material_embeddings_vector ON material_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Extend session_materials with extraction metadata (already in Story 5.x)
-- (No changes needed - content_extracted and extraction_status already exist)

-- Store extraction job history for monitoring and debugging
CREATE TABLE extraction_jobs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    material_id UUID NOT NULL REFERENCES session_materials(id) ON DELETE CASCADE,
    status VARCHAR(20) NOT NULL,                 -- PENDING, IN_PROGRESS, COMPLETED, FAILED
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    error_message TEXT,
    chunks_extracted INTEGER DEFAULT 0,
    embeddings_created INTEGER DEFAULT 0,
    execution_duration_ms BIGINT,                -- For performance monitoring
    lambda_request_id VARCHAR(100),              -- AWS Lambda request ID for tracing
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_extraction_jobs_material_id (material_id),
    INDEX idx_extraction_jobs_status (status)
);
```

## Acceptance Criteria

1. **AC1: Content Extraction Pipeline (Lambda)**
   - Lambda triggered when `SessionMaterialsUploadedEvent` occurs
   - Extract text from PDFs using AWS Textract (detect document text)
   - Extract text from PPTX using python-pptx library (slide content + notes)
   - Transcribe videos using AWS Transcribe (MP4, MOV, AVI formats)
   - Chunk extracted text: 500 tokens per chunk, 50-token overlap
   - Store chunks in `material_content_chunks` table
   - Update `session_materials.extraction_status = IN_PROGRESS → COMPLETED`
   - Failed extractions set `extraction_status = FAILED` with error message

2. **AC2: Embedding Generation Pipeline**
   - Generate embeddings for each chunk using Bedrock Titan Embeddings G1
   - Store 1024-dimension vectors in `material_embeddings` table
   - Create HNSW index for fast cosine similarity search
   - Batch process chunks (10 at a time) via SQS queue
   - Track embedding generation in `extraction_jobs` table

3. **AC3: RAG Search API**
   - Endpoint: `POST /api/v1/archive/search`
   - Request: `{ query: "digital transformation strategies", eventCode?: "bat-2025", topK: 10 }`
   - Retrieval: Convert query to embedding, search pgvector with cosine similarity
   - Augmentation: Retrieve top K relevant chunks with session context
   - Generation: Use Bedrock Claude 3.5 Sonnet to generate answer from context
   - Response: `{ answer: "...", sources: [{ sessionSlug, materialFileName, snippet, relevanceScore }] }`
   - Pagination: Support `page` and `limit` query params
   - Filtering: Filter by event code, date range, material type

4. **AC4: Search Results Display (Frontend)**
   - ArchivePage shows search bar prominently at top
   - Search bar has placeholder: "Search through all session materials..."
   - Debounced search (300ms delay) to reduce API calls
   - Loading indicator while search is in progress
   - Results display:
     - LLM-generated answer at top (if query is a question)
     - List of source materials with snippets
     - Event and session context for each result
     - Page/slide/timestamp reference where text was found
   - Empty state: "No results found for 'query'" with suggestions
   - Error handling: Network errors, timeout (10s), API errors

5. **AC5: Search Result Navigation**
   - "View Session" button navigates to EventProgram for that session
   - "Download Material" button opens CloudFront URL in new tab
   - Clicking result card highlights relevant text in material (future enhancement placeholder)
   - Back button returns to search results (maintains scroll position)

6. **AC6: Performance Requirements**
   - Search API response time: <2s for P95 (including LLM generation)
   - Vector search: <100ms for 1M embeddings (HNSW index optimization)
   - Content extraction: <5 minutes for 100-page PDF
   - Video transcription: <2× real-time (10-minute video → <20 minutes)
   - Frontend search: Debounced, no blocking UI

7. **AC7: Cost Optimization**
   - Use pgvector (free) instead of OpenSearch Serverless (~$175/month)
   - Cache embeddings (never regenerate unless content changes)
   - Batch embedding generation (reduce Bedrock API calls)
   - Use Bedrock On-Demand pricing (no provisioned throughput)
   - Set CloudWatch log retention to 30 days (reduce storage costs)
   - Estimated monthly cost: <$100 for 1,000 materials + 10K searches

8. **AC8: Extraction Status Monitoring**
   - Admin dashboard shows extraction pipeline status (future enhancement)
   - CloudWatch alarms for extraction failures (>5 failures/hour)
   - Retry logic for failed extractions (3 attempts with exponential backoff)
   - Manual re-extraction API: `POST /admin/materials/{materialId}/re-extract` (organizer-only)

## Dev Notes - Implementation Guide

### Quick Start
1. **Enable pgvector in PostgreSQL**: `CREATE EXTENSION vector;`
2. **Deploy Lambda functions**: Content extractor + embedding generator
3. **Backfill existing materials**: Run extraction pipeline on materials with `extraction_status = PENDING`
4. **Add search interface**: Integrate ArchiveSearchBar into ArchivePage

### Technical Design Notes

**RAG Pipeline Architecture**:
```
Upload Material → S3 Event → Lambda Extractor → [Textract/Transcribe] → Chunks (DB)
                                                              ↓
                                                  SQS Queue → Lambda Embedder
                                                              ↓
                                              Bedrock Titan → Embeddings (pgvector)

Search Query → Bedrock Titan (query embedding) → pgvector (cosine similarity)
                                                       ↓
                                  Top K chunks → Bedrock Claude (answer generation)
                                                       ↓
                                      Response (answer + sources)
```

**Content Extraction Strategy**:
- **PDFs**: AWS Textract DetectDocumentText (handles scanned PDFs with OCR)
- **PPTX**: python-pptx library (extract slide text + notes)
- **Videos**: AWS Transcribe (start job, poll for completion, download transcript JSON)
- **Chunking**: tiktoken with cl100k_base encoding (OpenAI's tokenizer)
  - 500 tokens per chunk
  - 50-token overlap between chunks
  - Preserve sentence boundaries (don't split mid-sentence)

**Embedding Generation**:
- Model: Bedrock Titan Text Embeddings G1
- Dimensions: 1024 (optimized for semantic search)
- Cost: $0.0001 per 1K input tokens
- Batch size: 10 chunks per API call (reduce latency)

**Vector Search with pgvector**:
```sql
-- Example similarity search query (cosine distance)
SELECT
    mc.content,
    mc.page_number,
    mc.slide_number,
    mc.timestamp_seconds,
    sm.file_name,
    sm.material_type,
    s.session_slug,
    s.title AS session_title,
    e.event_code,
    1 - (me.embedding <=> :query_embedding) AS similarity_score
FROM material_embeddings me
JOIN material_content_chunks mc ON me.chunk_id = mc.id
JOIN session_materials sm ON mc.material_id = sm.id
JOIN sessions s ON sm.session_id = s.id
JOIN events e ON s.event_id = e.id
WHERE e.event_code = :event_code_filter  -- Optional filter
ORDER BY me.embedding <=> :query_embedding  -- Cosine distance (HNSW index)
LIMIT 10;
```

**Database Transaction Boundaries**:
- Chunk storage: Single transaction per material (all chunks atomically)
- Embedding generation: Separate transaction per batch (10 chunks)
- Search query: Read-only transaction (no writes)

**Caching Strategy**:
- Embeddings: Never regenerate (cached forever unless content changes)
- Search results: React Query with 5-minute stale time
- LLM responses: Not cached (answers are context-dependent)

**Error Handling Patterns**:
- Lambda timeout: Increase to 15 minutes for large files
- Textract/Transcribe errors: Retry 3 times with exponential backoff
- Bedrock throttling: Implement exponential backoff with jitter
- pgvector search timeout: Set statement_timeout to 10s

**Performance Considerations**:
- HNSW index tuning: `m=16` (neighbors), `ef_construction=64` (build quality)
- Vector search: <100ms for 1M embeddings with HNSW
- Embedding generation: Batch 10 chunks → 1 API call (reduce latency)
- LLM generation: Stream response to frontend (progressive rendering)

### Cost Analysis

**Monthly Cost Estimate** (1,000 materials, 10K searches):

**Content Extraction** (one-time per material):
- AWS Textract: 500 PDFs × 10 pages avg × $0.0015/page = $7.50
- AWS Transcribe: 100 videos × 10 min avg × $0.024/min = $24.00
- **Total**: ~$32/month (decreases over time as materials are processed)

**Embedding Generation** (one-time per material):
- Bedrock Titan: 1,000 materials × 20 chunks avg × 200 tokens/chunk × $0.0001/1K = $0.40
- **Total**: ~$1/month (decreases over time)

**Search Queries** (recurring):
- Bedrock Titan (query embedding): 10K searches × 50 tokens avg × $0.0001/1K = $0.05
- Bedrock Claude (answer generation): 10K searches × 1K input tokens × $3/1M = $30.00
- Bedrock Claude (output): 10K searches × 200 output tokens × $15/1M = $30.00
- **Total**: ~$60/month (recurring)

**Storage**:
- pgvector (RDS): Included in existing PostgreSQL costs (no additional charge)
- S3 materials: Already included from Story 5.x
- CloudWatch logs: ~$5/month (30-day retention)
- **Total**: ~$5/month

**Grand Total**: ~$98/month (stable state after initial extraction)

**Cost Optimization Recommendations**:
- Use pgvector (free) instead of OpenSearch Serverless ($175/month)
- Cache embeddings forever (never regenerate)
- Batch Bedrock API calls (reduce per-request overhead)
- Set CloudWatch log retention to 30 days (vs 90 days = $15/month savings)
