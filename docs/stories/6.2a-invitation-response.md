## Status

## Story

**As a** speaker,
**I want** to respond to event invitations (Accept/Decline/Tentative) via a simple web form accessed through my invitation email link,
**so that** I can confirm my participation without email back-and-forth with organizers.

## Domain Context

### Primary Domain

**Speaker Coordination Domain** - Self-service invitation response workflow

This story creates the speaker-facing response interface that allows speakers to Accept, Decline, or mark Tentative for event invitations. It consumes the magic link token infrastructure from Story 6.1a and transitions speaker workflow states.

### Involved Services

**Services:**
- **Event Management Service**: Response processing, workflow transitions, organizer notifications
- **API Gateway**: Routes `/api/v1/speaker-portal/**` endpoints (public, token-authenticated)
- **Company User Management Service**: User data enrichment for response confirmation

**Shared-Kernel Foundation** (automatically included):
- `SpeakerWorkflowState` enum (INVITED → ACCEPTED/DECLINED)
- Domain events (`SpeakerResponseReceivedEvent`)
- ErrorResponse (standardized error format)

## Architecture Context

### Architecture Patterns

**Referenced from architecture docs (SOURCE OF TRUTH):**

1. **6-State Workflow Model** (from architecture plan):
   - INVITED → ACCEPTED (on Accept)
   - INVITED → DECLINED (on Decline, terminal)
   - INVITED (is_tentative=true) (on Tentative, awaits follow-up)

2. **Token Consumption**:
   - RESPOND tokens are single-use for Accept/Decline
   - Tentative does NOT consume token (speaker may change mind)

3. **ADR-003** (Meaningful Identifiers):
   - Response returns `eventCode`, `username` (not UUIDs)

4. **ADR-004** (Cross-Service References):
   - Event/Session data fetched via internal lookups
   - User enrichment via HTTP client

### API Specification

**From Architecture Plan (Source of Truth):**

```yaml
POST /api/v1/speaker-portal/respond
  Description: Submit speaker response to invitation
  Auth: Magic link token (no session auth)

  Request Body:
    token: string           # Magic link token from email
    response: enum          # "ACCEPT" | "DECLINE" | "TENTATIVE"
    reason?: string         # Required if DECLINE or TENTATIVE
    constraints?: string    # Optional: travel, schedule constraints
    preferences?:           # Optional: shown on Accept
      timeSlot?: string     # "morning" | "afternoon" | "no_preference"
      travelRequirements?: string  # "local" | "accommodation" | "virtual"
      technicalRequirements?: string[]  # ["mac_adapter", "remote_option"]
      initialTitle?: string # Preliminary presentation title
      comments?: string     # Message to organizer

  Response 200 OK:
    speakerName: string
    eventName: string
    eventDate: string       # Formatted: "20. November 2025"
    sessionTitle?: string   # If session assigned
    nextSteps: string[]     # ["Complete your profile", "Submit title and abstract"]
    contentDeadline?: string # If accepted
    dashboardUrl?: string   # Link to speaker dashboard

  Response 400 Bad Request:
    error: "REASON_REQUIRED"  # When DECLINE/TENTATIVE without reason

  Response 401 Unauthorized:
    error: "TOKEN_EXPIRED" | "TOKEN_USED" | "TOKEN_NOT_FOUND"

  Response 409 Conflict:
    error: "ALREADY_RESPONDED"
    previousResponse: string  # "ACCEPTED" | "DECLINED"
    respondedAt: string
```

## Acceptance Criteria

### AC1: Token Validation on Page Load

1. When speaker clicks magic link, frontend calls `POST /speaker-portal/validate-token`
2. Valid token: Display invitation details and response form
3. Expired token: Show "Link expired" message with contact info
4. Used token (RESPOND): Show "Already responded" with previous response
5. Invalid token: Show "Invalid link" error page

### AC2: Response Form Display

1. Display event details: title, date, location, format, expected attendance
2. Display session details (if assigned): title, description, time slot
3. Show "Why we chose you" message from invitation (if present)
4. Show response deadline with countdown ("10 days remaining")
5. Three response buttons: Accept, Decline, Tentative (styled per wireframe)

### AC3: Accept Response Flow

1. Clicking "Accept" reveals preferences form (conditional display)
2. Preferences form includes: time slot preference, travel requirements, technical needs, initial title, comments
3. All preference fields are optional
4. Submit calls `POST /speaker-portal/respond` with `response: "ACCEPT"`
5. Token is marked as used (single-use enforcement)
6. `workflow_state` transitions to `ACCEPTED`
7. `accepted_at` timestamp is set
8. Success page shows next steps and content deadline

### AC4: Decline Response Flow

1. Clicking "Decline" shows reason input (required field)
2. Optional: Alternative speaker suggestion field
3. Submit calls `POST /speaker-portal/respond` with `response: "DECLINE"`
4. Reason is required - validation error if empty
5. Token is marked as used (single-use enforcement)
6. `workflow_state` transitions to `DECLINED` (terminal state)
7. `declined_at` timestamp and `decline_reason` are set
8. Thank you page with option to join mailing list

### AC5: Tentative Response Flow

1. Clicking "Tentative/Maybe" shows reason input (required field)
2. Reason examples: "Need to check travel dates", "Awaiting budget approval"
3. Submit calls `POST /speaker-portal/respond` with `response: "TENTATIVE"`
4. Token is NOT consumed (speaker can return and change response)
5. `workflow_state` stays `INVITED`, `is_tentative` set to `true`
6. `tentative_reason` is stored
7. Confirmation page explains organizer will follow up

### AC6: Organizer Notification

1. On any response, emit `SpeakerResponseReceivedEvent` domain event
2. Send email notification to event organizer(s)
3. Email includes: speaker name, response type, reason (if provided), link to speaker management
4. Notification sent asynchronously (non-blocking)

### AC7: Already Responded Handling

1. If speaker returns with used token, show previous response
2. Display: "You responded [ACCEPTED/DECLINED] on [date]"
3. For ACCEPTED: Show link to speaker dashboard
4. For DECLINED: Show option to contact organizer to reconsider
5. For TENTATIVE: Allow updating response (token still valid)

### AC8: Mobile Responsiveness

1. Response form works on mobile devices (375px width minimum)
2. Buttons stack vertically on small screens
3. Preferences form slides in smoothly
4. Touch-friendly button sizes (min 44px tap target)

## Dev Notes - Implementation Guide

### Response Processing Logic

```java
@Service
@RequiredArgsConstructor
public class SpeakerResponseService {

    private final SpeakerPoolRepository speakerPoolRepository;
    private final MagicLinkService magicLinkService;
    private final DomainEventPublisher eventPublisher;
    private final OrganizerNotificationService notificationService;

    @Transactional
    public ResponseResult processResponse(SpeakerResponseRequest request) {
        // 1. Validate token
        TokenValidationResult tokenResult = magicLinkService.validateToken(request.token());
        if (!tokenResult.valid()) {
            throw new InvalidTokenException(tokenResult.error());
        }

        // 2. Get speaker pool entry
        SpeakerPool speaker = speakerPoolRepository.findById(tokenResult.speakerPoolId())
            .orElseThrow(() -> new SpeakerNotFoundException(tokenResult.speakerPoolId()));

        // 3. Check if already responded (for ACCEPT/DECLINE)
        if (speaker.getWorkflowState() != SpeakerWorkflowState.INVITED) {
            throw new AlreadyRespondedException(speaker.getWorkflowState(), speaker.getAcceptedAt());
        }

        // 4. Process based on response type
        switch (request.response()) {
            case ACCEPT -> processAccept(speaker, request);
            case DECLINE -> processDecline(speaker, request);
            case TENTATIVE -> processTentative(speaker, request);
        }

        // 5. Publish domain event
        eventPublisher.publish(new SpeakerResponseReceivedEvent(
            speaker.getId(),
            speaker.getUsername(),
            speaker.getEventCode(),
            request.response(),
            request.reason(),
            Instant.now()
        ));

        // 6. Send organizer notification (async)
        notificationService.notifyOrganizerOfResponse(speaker, request.response());

        // 7. Return result
        return buildResponseResult(speaker, request.response());
    }

    private void processAccept(SpeakerPool speaker, SpeakerResponseRequest request) {
        speaker.setWorkflowState(SpeakerWorkflowState.ACCEPTED);
        speaker.setAcceptedAt(Instant.now());
        speaker.setIsTentative(false);

        if (request.preferences() != null) {
            speaker.setPreferences(request.preferences());
        }

        speakerPoolRepository.save(speaker);

        // Consume token (single-use for ACCEPT)
        magicLinkService.markTokenAsUsed(request.token());
    }

    private void processDecline(SpeakerPool speaker, SpeakerResponseRequest request) {
        if (request.reason() == null || request.reason().isBlank()) {
            throw new ValidationException("Reason is required for decline");
        }

        speaker.setWorkflowState(SpeakerWorkflowState.DECLINED);
        speaker.setDeclinedAt(Instant.now());
        speaker.setDeclineReason(request.reason());

        speakerPoolRepository.save(speaker);

        // Consume token (single-use for DECLINE)
        magicLinkService.markTokenAsUsed(request.token());
    }

    private void processTentative(SpeakerPool speaker, SpeakerResponseRequest request) {
        if (request.reason() == null || request.reason().isBlank()) {
            throw new ValidationException("Reason is required for tentative response");
        }

        // State stays INVITED, but mark as tentative
        speaker.setIsTentative(true);
        speaker.setTentativeReason(request.reason());

        speakerPoolRepository.save(speaker);

        // Do NOT consume token - speaker may return to change response
    }
}
```

### Frontend Response Page Structure

```typescript
// InvitationResponsePage.tsx
export function InvitationResponsePage() {
  const [searchParams] = useSearchParams();
  const token = searchParams.get('token');

  const { data: invitation, error, isLoading } = useQuery({
    queryKey: ['invitation', token],
    queryFn: () => speakerPortalService.validateToken(token!),
    enabled: !!token,
  });

  const respondMutation = useMutation({
    mutationFn: speakerPortalService.respond,
    onSuccess: (data) => {
      // Show success state
    },
  });

  if (error) {
    return <TokenErrorState error={error} />;
  }

  if (respondMutation.isSuccess) {
    return <ResponseConfirmation result={respondMutation.data} />;
  }

  return (
    <SpeakerPortalLayout>
      <EventDetailsCard invitation={invitation} />
      <ResponseForm
        invitation={invitation}
        onSubmit={(response) => respondMutation.mutate({ token, ...response })}
        isSubmitting={respondMutation.isPending}
      />
    </SpeakerPortalLayout>
  );
}
```
