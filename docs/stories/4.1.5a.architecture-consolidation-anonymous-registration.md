### Problem Statement
Current stories 4.1.5 and 4.1.6 contain architectural inconsistencies that must be resolved before implementation:

1. **Wrong Endpoints**: Stories reference `/api/v1/registrations` instead of `/api/v1/events/{eventCode}/registrations`
2. **Wrong Registration Scope**: Stories suggest session registration instead of whole-event registration
3. **Data Duplication**: registrations table duplicates user data (attendee_name, attendee_email) that exists in user_profiles
4. **No Anonymous Support**: user_profiles.cognito_id is NOT NULL, preventing anonymous registration
5. **Missing ADR**: No documented architecture decision for anonymous registration pattern
6. **Wireframe Misalignment**: Stories don't match wireframes for registration flow

### Cross-Service Architecture Pattern

This consolidation establishes a clean microservices architecture:

```
┌─────────────────────────────────────────────┐
│ Event Management Service                    │
│                                             │
│  registrations table:                       │
│  - id, event_id, registration_code         │
│  - attendee_username (ref to user_profiles)│
│  - status, registration_date               │
│                                             │
│  RegistrationService:                       │
│  1. Receive registration request           │
│  2. Call UserManagementClient (existing)    │
│  3. Create/find user_profile               │
│  4. Create registration with username ref   │
└─────────────────┬───────────────────────────┘
                  │ HTTP API Call
                  │ (via UserManagementClient)
                  ▼
┌─────────────────────────────────────────────┐
│ Company User Management Service             │
│                                             │
│  user_profiles table:                       │
│  - id, username, email                      │
│  - cognito_user_id (NULLABLE)              │
│  - first_name, last_name, company          │
│                                             │
│  UserService:                               │
│  - Create anonymous user (cognito_id=NULL) │
│  - Link account on first Cognito login     │
└─────────────────────────────────────────────┘
```

### Benefits of This Architecture

✅ **Single Source of Truth**: User data lives only in user_profiles
✅ **No Data Duplication**: registrations reference users, don't duplicate
✅ **Microservices Independence**: Services communicate via APIs, not shared DB
✅ **Clean Domain Boundaries**: Event Service = events/registrations, User Service = users
✅ **Scalable**: Can cache user lookups, batch queries prevent N+1
✅ **Reuse Existing Infrastructure**: Leverages existing UserManagementClient from speaker integration

### Whole-Event Registration (Not Sessions)

**Decision**: Registrations are for the whole event, not individual sessions.

**Rationale**:
- Matches wireframes (no session selection shown)
- Matches existing data model (registrations.UNIQUE(event_id, attendee_username))
- Simpler user experience
- Session preferences can be tracked separately if needed for capacity planning (future story)

### Anonymous to Authenticated Flow

**Anonymous Registration** (Story 4.1.5):
1. User registers without creating account
2. user_profile created with cognito_id=NULL
3. Email is unique identifier

**Account Creation** (Story 4.1.6+):
1. User decides to create account from confirmation page
2. Redirected to signup with email pre-filled
3. User completes Cognito registration

**Auto-Linking** (Post-authentication):
1. Cognito post-authentication trigger fires on first login
2. Finds user_profile by email where cognito_id IS NULL
3. Sets cognito_id on existing user_profile
4. User now has authenticated account with previous registration history
