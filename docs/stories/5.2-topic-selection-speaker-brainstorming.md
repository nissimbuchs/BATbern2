# Story 5.2: Topic Selection & Speaker Brainstorming

## Status
Ready for Review (Backend ✅, Frontend ✅, Infrastructure ✅, Integration Tests 89-100%, Frontend Tests Deferred)

## Story

**As a** organizer,
**I want** to select topics from our backlog with intelligent suggestions and brainstorm potential speakers,
**so that** I can plan compelling events with appropriate speaker assignments.

## Domain Context

### Primary Domain

**Event Management Domain** - Organizer workflows and event planning automation

This story implements the first three steps of the 16-step event workflow (FR2):
- **Step 1-2**: Topic Selection with heat map visualization, staleness detection, and similarity scoring
- **Step 2-3**: Speaker Brainstorming with speaker pool management and assignment strategy

## Architecture Context

### Architecture Patterns

**Data Models** (03-data-architecture.md):
- **Topic Entity**: id (UUID PK), title, description, category, staleness_score (0-100), usage_history (JSONB), last_used_date, similarity_scores (JSONB)
- **TopicUsageHistory Entity**: id (UUID PK), topic_id (FK), event_id (FK), used_date, attendance, engagement_score
- **SpeakerPool Entity**: id (UUID PK), event_id (FK), speaker_name, company, expertise, assigned_organizer_id (FK), status (EventWorkflowState), notes
- **Event Entity** (from Story 5.1): workflow_state column (enum → VARCHAR via converter)

**API Design** (04-api-design.md, 04-api-event-management.md):
- RESTful endpoints: `GET /api/v1/topics`, `POST /api/v1/events/{id}/speakers/pool`
- OpenAPI 3.1 specification with JSON filter parameter (MongoDB-style)
- JWT authentication via AWS Cognito with role-based access (`ORGANIZER` role required)

**Frontend** (05-frontend-architecture.md):
- React 19 + TypeScript with Material-UI components
- **i18n**: react-i18next with namespace-based translations
  - **Namespaces**: `organizer.json` for organizer-specific UI, `common.json` for shared UI
  - **Primary Language**: German (de-CH), fallback: English (en-US)
  - **Translation Keys**: Nested JSON structure with type-safe keys (e.g., `t('organizer.topicBacklog.title')`)
- State management: Zustand for local state, React Query for server state
- Recharts for heat map visualization

**Backend** (06-backend-architecture.md):
- Spring Boot 3.2 + Java 21 with DDD layered architecture
- Service layer implements business logic (TopicService, SpeakerPoolService)
- Repository layer uses JPQL constructor projections for DTO optimization (avoid N+1 queries)
- Integration with EventWorkflowStateMachine from Story 5.1a for state transitions

## Acceptance Criteria

From Epic 5, Story 5.2:

**Topic Selection (Step 1-2):**
1. **Topic Backlog**: Display searchable list of all available topics
2. **Heat Map Visualization**: Display topic usage heat map showing frequency over last 24 months
3. **Color-Coded Freshness**: Topics color-coded by staleness (red: <6 months, yellow: 6-12 months, green: >12 months)
4. **Similarity Scoring**: Calculate similarity scores between topics using TF-IDF and cosine similarity
5. **Duplicate Warnings**: Automatic warnings when similarity score >70% with recent topics
6. **Staleness Score**: Display 0-100 staleness score (100 = safe to reuse, 0 = too recent)
7. **Override Capability**: Allow organizers to override warnings with justification
8. **Topic Creation**: Allow new topic creation with description inline

**Speaker Brainstorming (Step 2-3):**
9. **Speaker Pool**: Organizers can add potential speakers to event (name, company, expertise)
10. **Speaker Notes**: Free-text notes field for each potential speaker
11. **Assignment Strategy**: Assign speakers to specific organizers for outreach
12. **Contact Distribution**: Track which organizer will contact which speaker
13. **Speaker Status**: Initial status = "OPEN" (not yet contacted)

**Workflow Engine Integration:**
14. **Event State Transition**: When topic selected, call `eventWorkflowStateMachine.transitionToState(eventId, EventWorkflowState.TOPIC_SELECTION, organizerId)`
15. **Speaker Pool State**: When speaker added to pool, call `speakerWorkflowService.updateSpeakerWorkflowState(sessionId, speakerId, SpeakerWorkflowState.IDENTIFIED, organizerId)`
16. **Validation**: Ensure event can transition to TOPIC_SELECTION state before allowing topic selection

**Technical Implementation:**
17. **Topic Entity Enhancement**: Add staleness tracking, similarity scores, usage history
18. **Speaker Pool Table**: event_speaker_pool (event_id, speaker_name, company, expertise, assigned_organizer_id, status, notes)
19. **REST API**: GET /api/topics, POST /api/events/{id}/speakers/pool
20. **React Components**: TopicSelector, SpeakerBrainstormingPanel
21. **Domain Events**: TopicSelectedEvent, SpeakerAddedToPoolEvent

## Dev Notes - Implementation Guide

### Technical Design Notes

**Similarity Calculation Algorithm (AC4):**
- **TF-IDF (Term Frequency-Inverse Document Frequency)**:
  - Calculate term frequency for each word in topic title + description
  - Calculate inverse document frequency across all topics
  - Compute TF-IDF vector for each topic
- **Cosine Similarity**:
  - Compute cosine similarity between TF-IDF vectors: `similarity = (A · B) / (||A|| × ||B||)`
  - Threshold: >70% similarity triggers duplicate warning
- **Implementation**: `SimilarityCalculationService.calculateSimilarity(topic1, topic2)`
- **Storage**: Store similarity scores in `topics.similarity_scores` JSONB column: `{"topicId": "abc", "score": 0.85}`

**Staleness Score Algorithm (AC6):**
- **Calculation Logic**:
  - 0-100 scale where 100 = safe to reuse (>12 months since last use)
  - Formula: `stalenessScore = min(100, (monthsSinceLastUse / 12) * 100)`
  - Examples: 0 months = score 0, 6 months = score 50, 12 months = score 100, 24 months = score 100
- **Implementation**: `StalenessScoreService.calculateStaleness(topic)` called on daily scheduled job
- **Storage**: Store in `topics.staleness_score` INT column
- **Color Coding**: Red (<50), Yellow (50-83), Green (>83)

// web-frontend/src/services/topicService.ts
import { apiClient } from '@/services/api/apiClient';
import type { Topic, TopicFilters, TopicListResponse, CreateTopicRequest } from '@/types/topic.types';

class TopicService {
  /**
   * Get topics with filtering (Story 5.2)
   * Follows pattern from EventTypeService and WorkflowService
   */
  async getTopics(filters: TopicFilters): Promise<TopicListResponse> {
    const response = await apiClient.get('/api/v1/topics', {
      params: {
        filter: JSON.stringify(filters),
        sort: filters.sort,
        page: filters.page,
        limit: filters.limit,
        include: filters.include
      }
    });
    return response.data;
  }

  async getTopicById(id: string, include?: string): Promise<Topic> {
    const response = await apiClient.get(`/api/v1/topics/${id}`, {
      params: { include }
    });
    return response.data;
  }

  async createTopic(request: CreateTopicRequest): Promise<Topic> {
    const response = await apiClient.post('/api/v1/topics', request);
    return response.data;
  }

  async selectTopicForEvent(eventId: string, topicId: string, justification?: string): Promise<void> {
    await apiClient.post(`/api/v1/events/${eventId}/topics`, {
      topicId,
      justification
    });
  }
}
