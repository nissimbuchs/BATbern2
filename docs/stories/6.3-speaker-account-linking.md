## Status

Draft

**Created:** 2026-01-19

## Story

**As an** external speaker who accepted an invitation,
**I want** to create an account and have it automatically linked to my speaker participation,
**so that** I can access the speaker dashboard to manage my sessions and submissions.

## Problem Statement

Currently, external speakers (those without existing BATbern accounts) can:
- ✅ Receive invitation emails
- ✅ Accept/decline via token-based response portal
- ✅ Submit preferences (time slots, presentation title)

However, after accepting:
- ❌ No prompt to create an account
- ❌ `SpeakerPool.username` remains NULL
- ❌ Cannot access speaker dashboard
- ❌ Organizers must manually link speakers to accounts

This creates a broken user journey where accepted speakers have no way to manage their participation.

### Acceptance Criteria

#### AC1: Account Creation Prompt on Acceptance
**Given** an external speaker has accepted an invitation via the response portal
**When** the acceptance confirmation is displayed
**Then** a prominent "Create Account" call-to-action is shown with pre-filled email

#### AC2: Pre-filled Registration Form
**Given** a speaker clicks "Create Account" from the acceptance confirmation
**When** the registration page loads
**Then** the email field is pre-filled with the speaker's invitation email
**And** a message indicates this will link to their speaker participation

#### AC3: Automatic SpeakerPool Linking on Registration
**Given** a new user registers with an email matching a SpeakerPool entry
**When** the Cognito post-confirmation trigger executes
**Then** the `speaker_pool.username` field is updated with the new username
**And** the speaker can immediately access the speaker dashboard

#### AC4: Automatic Speaker Entity Creation
**Given** a SpeakerPool entry is linked to a new user account
**When** the linking process completes
**Then** a Speaker entity is created if one doesn't exist for that username
**And** the Speaker entity has default values for expertise areas and topics

#### AC5: Multiple Event Support
**Given** an external speaker has accepted invitations for multiple events
**When** they create an account
**Then** ALL matching SpeakerPool entries (by email) are linked to their username
**And** they can see all their speaker participations in the dashboard

#### AC6: Manual Linking Fallback
**Given** automatic linking fails or speaker uses different email
**When** an organizer needs to link a speaker manually
**Then** an API endpoint allows linking SpeakerPool to username
**And** the endpoint validates the username exists in User Service

#### AC7: Idempotent Linking
**Given** a SpeakerPool entry is already linked to a username
**When** the linking process runs again (e.g., Lambda retry)
**Then** no error occurs and the existing link is preserved

## Architecture Context

### Architecture Decision: Lambda Direct DB Write + Event Publishing

**Decision:** Extend the existing Cognito post-confirmation Lambda to directly write to `speaker_pool` table, then publish an event for Speaker entity creation.

**Rationale:** The Lambda already crosses service boundaries by writing directly to `user_profiles` and `role_assignments` tables. This pattern provides:
- **Atomicity**: SpeakerPool linking happens in the same transaction as user creation
- **Reliability**: No dependency on Event Management Service availability during registration
- **Immediacy**: Speaker sees linked state immediately after registration
- **Consistency**: Follows established Lambda pattern in codebase

**Alternatives Considered:**
1. ❌ **HTTP API Call**: Would add latency and failure modes to registration critical path
2. ❌ **Async Event Only**: Would result in eventually consistent state (speaker briefly sees unlinked)

**Mitigation for Service Boundary Concern:**
- Lambda publishes `SpeakerPoolLinked` event to EventBridge after DB write
- Event Management Service listens and creates Speaker entity (business logic stays in service)
- Schema dependency documented in Lambda code

### Architecture Patterns

**Referenced from architecture docs (SOURCE OF TRUTH):**

1. **ADR-003** (Meaningful Identifiers in Public APIs):
   - SpeakerPool stores `username` (meaningful ID) when linked
   - API endpoints use username for speaker identification

2. **ADR-004** (Factor User Fields from Domain Entities):
   - SpeakerPool stores contact fields for unlinked speakers (exemption documented)
   - Once linked, profile data fetched from User Service via HTTP

3. **Existing Auto-Linking Pattern** (Attendee Registration):
   - Cognito post-confirmation Lambda checks for email matches
   - Updates existing records to link with new user account
   - Same pattern to be extended for SpeakerPool

4. **Event-Driven Downstream Processing** (NEW):
   - Lambda publishes domain event after DB write
   - Event Management Service creates Speaker entity via event listener
   - Decouples data linking from business entity creation

### Component 2: Cognito Lambda - SpeakerPool Linking with Event Publishing

**File:** `infrastructure/lib/lambda/triggers/post-confirmation.ts`

Extend existing logic to link SpeakerPool entries and publish event for downstream processing:

```typescript
import { EventBridgeClient, PutEventsCommand } from '@aws-sdk/client-eventbridge';

const eventBridgeClient = new EventBridgeClient({ region: process.env.AWS_REGION });

interface SpeakerPoolLinkResult {
  linkedCount: number;
  linkedEntries: Array<{ id: string; eventId: string }>;
}

/**
 * Link speaker pool entries to newly registered user.
 * Story 6.3: Speaker Account Creation and Linking
 *
 * ARCHITECTURE NOTE: This follows the established pattern of the Lambda
 * directly updating domain tables during user creation. The existing Lambda
 * already writes to user_profiles and role_assignments tables.
 *
 * An event is published for Event Management Service to create the Speaker
 * entity (which requires service-level business logic).
 */
async function linkSpeakerPoolEntries(
  client: PoolClient,
  email: string,
  username: string
): Promise<SpeakerPoolLinkResult> {
  // Case-insensitive email match, only unlinked entries
  // AC5: Links ALL matching entries (multiple events)
  // AC7: WHERE username IS NULL ensures idempotency
  const result = await client.query(
    `UPDATE speaker_pool
     SET username = $1, updated_at = NOW()
     WHERE LOWER(email) = LOWER($2)
       AND username IS NULL
     RETURNING id, event_id`,
    [username, email]
  );

  const linkedEntries = result.rows.map(row => ({
    id: row.id,
    eventId: row.event_id
  }));

  console.log('Speaker pool linking result', {
    username,
    email,
    linkedCount: result.rowCount,
    linkedEntries
  });

  return {
    linkedCount: result.rowCount || 0,
    linkedEntries
  };
}

/**
 * Publish SpeakerPoolLinked event to EventBridge.
 * Event Management Service listens and creates Speaker entity.
 */
async function publishSpeakerPoolLinkedEvent(
  username: string,
  email: string,
  linkedEntries: Array<{ id: string; eventId: string }>
): Promise<void> {
  const event = {
    Source: 'batbern.cognito',
    DetailType: 'SpeakerPoolLinked',
    Detail: JSON.stringify({
      username,
      email,
      linkedCount: linkedEntries.length,
      linkedEntries,
      timestamp: new Date().toISOString()
    }),
    EventBusName: process.env.EVENT_BUS_NAME || 'default'
  };

  try {
    await eventBridgeClient.send(new PutEventsCommand({ Entries: [event] }));
    console.log('Published SpeakerPoolLinked event', { username, linkedCount: linkedEntries.length });
  } catch (error) {
    // Log but don't fail - event publishing is best-effort
    // Manual linking fallback (AC6) handles cases where Speaker entity isn't created
    console.error('Failed to publish SpeakerPoolLinked event', { error, username });
  }
}

// In createUser() function, after user creation and BEFORE commit:
const { linkedCount, linkedEntries } = await linkSpeakerPoolEntries(client, email, username);

await client.query('COMMIT');

// Publish event AFTER commit (ensures data is persisted)
if (linkedCount > 0) {
  await publishSpeakerPoolLinkedEvent(username, email, linkedEntries);
  await publishMetric('SpeakerPoolEntriesLinked', linkedCount, 'Count');
}
```

### Component 2c: Event Management Service Listener

**File:** `services/event-management-service/src/main/java/ch/batbern/events/listener/SpeakerPoolLinkedEventListener.java`

```java
package ch.batbern.events.listener;

import ch.batbern.events.service.SpeakerService;
import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import io.awspring.cloud.sqs.annotation.SqsListener;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

/**
 * Listens for SpeakerPoolLinked events from Cognito Lambda.
 * Creates Speaker entity when external speaker registers.
 * Story 6.3: Speaker Account Creation and Linking
 */
@Component
@RequiredArgsConstructor
@Slf4j
public class SpeakerPoolLinkedEventListener {

    private final SpeakerService speakerService;
    private final ObjectMapper objectMapper;

    /**
     * Handle SpeakerPoolLinked event from EventBridge (via SQS).
     * Creates Speaker entity if it doesn't exist.
     *
     * Idempotent: Safe to process same event multiple times.
     */
    @SqsListener("${app.sqs.speaker-pool-linked-queue:speaker-pool-linked-queue}")
    public void handleSpeakerPoolLinked(String message) {
        try {
            JsonNode event = objectMapper.readTree(message);
            JsonNode detail = event.path("detail");

            String username = detail.path("username").asText();
            int linkedCount = detail.path("linkedCount").asInt();

            log.info("Processing SpeakerPoolLinked event: username={}, linkedCount={}",
                username, linkedCount);

            // Create Speaker entity if doesn't exist (AC4)
            // This is idempotent - safe for retries
            speakerService.ensureSpeakerExists(username);

            log.info("Successfully processed SpeakerPoolLinked for {}", username);

        } catch (Exception e) {
            log.error("Failed to process SpeakerPoolLinked event", e);
            // Let SQS retry or send to DLQ
            throw new RuntimeException("Failed to process SpeakerPoolLinked event", e);
        }
    }
}
```

**Infrastructure:** EventBridge rule routes `SpeakerPoolLinked` events to SQS queue consumed by Event Management Service.

### Component 3: Backend - Manual Linking Endpoint

**File:** `services/event-management-service/src/main/java/ch/batbern/events/service/SpeakerPoolService.java`

```java
@Transactional
public SpeakerPoolResponse linkSpeakerPoolToUser(
    String eventCode,
    UUID speakerPoolId,
    String username
) {
    // 1. Validate event exists
    Event event = eventRepository.findByEventCode(eventCode)
        .orElseThrow(() -> new EventNotFoundException(eventCode));

    // 2. Get speaker pool entry
    SpeakerPool speakerPool = speakerPoolRepository.findById(speakerPoolId)
        .orElseThrow(() -> new SpeakerPoolNotFoundException(speakerPoolId));

    // 3. Validate belongs to event
    if (!speakerPool.getEventId().equals(event.getId())) {
        throw new IllegalArgumentException(
            "Speaker pool entry does not belong to event " + eventCode);
    }

    // 4. Check not already linked to different user
    if (speakerPool.getUsername() != null
        && !speakerPool.getUsername().equals(username)) {
        throw new ConflictException(
            "Speaker pool already linked to user: " + speakerPool.getUsername());
    }

    // 5. Validate username exists (HTTP call to User Service)
    userApiClient.getUserByUsername(username); // Throws if not found

    // 6. Update speaker pool
    speakerPool.setUsername(username);
    SpeakerPool saved = speakerPoolRepository.save(speakerPool);

    // 7. Ensure Speaker entity exists
    speakerService.ensureSpeakerExists(username);

    log.info("Linked speaker pool {} to user {}", speakerPoolId, username);

    return SpeakerPoolResponse.fromEntity(saved);
}
```

### Component 4: Speaker Entity Creation

**File:** `services/event-management-service/src/main/java/ch/batbern/events/service/SpeakerService.java`

@Transactional
public Speaker ensureSpeakerExists(String username) {
    return speakerRepository.findByUsername(username)
        .orElseGet(() -> {
            Speaker speaker = Speaker.builder()
                .username(username)
                .workflowState(SpeakerWorkflowState.ACCEPTED)
                .expertiseAreas(new ArrayList<>())
                .speakingTopics(new ArrayList<>())
                .build();

            log.info("Created Speaker entity for {}", username);
            return speakerRepository.save(speaker);
        });
}

- Speaker role auto-assignment (remains ATTENDEE, organizer promotes)
