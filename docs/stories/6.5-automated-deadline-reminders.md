# Story 6.5: Automated Deadline Reminders

Status: dev-complete

## Story

As an **organizer**,
I want automated deadline reminders sent to speakers based on configurable rules,
so that I don't have to manually follow up on response and content submission deadlines.

## Domain Context

### Primary Domain

Speaker Coordination Domain - Automated deadline monitoring and reminder delivery

### Involved Services

- **Event Management Service**: Scheduler job, reminder service, email template rendering, reminder log persistence
- **Shared Kernel**: `EmailService` for AWS SES email delivery
- **API Gateway**: Route for organizer reminder configuration endpoints

### Cross-Domain Dependencies

- Story 6.1a (Magic Link Infrastructure) - Token generation for portal links in reminder emails
- Story 6.1b (Speaker Invitation System) - Email template patterns, `SpeakerInvitationEmailService` pattern
- Story 6.4 (Speaker Dashboard) - Dashboard link included in reminder emails
- Story 5.3 (Speaker Outreach Tracking) - `OutreachHistory` entity for logging reminders

## Acceptance Criteria

### AC1: Reminder Rule Configuration

1. Default reminder rules pre-configured (no UI needed for MVP):
   - **Tier 1 (Friendly)**: 14 days before deadline
   - **Tier 2 (Urgent)**: 7 days before deadline
   - **Tier 3 (Final)**: 3 days before deadline
2. Rules apply to both response deadlines (INVITED speakers) and content deadlines (ACCEPTED speakers)
3. Rules stored in application configuration (`application.yml`) - database-backed rules deferred to Phase 3
4. Each tier uses a distinct email template with appropriate tone escalation

### AC2: Automated Deadline Detection & Sending

1. Scheduler runs daily at 8:00 AM (before the 9 AM registration reminder job)
2. Uses ShedLock for distributed coordination (only ONE instance sends reminders)
3. Detects speakers with:
   - **Response deadline approaching**: `status = INVITED` AND `responseDeadline` within tier window
   - **Content deadline approaching**: `status = ACCEPTED` AND `contentStatus = PENDING` AND `contentDeadline` within tier window
4. Sends appropriate tier email based on days remaining until deadline
5. Includes magic link to speaker portal (dashboard link with VIEW token)

### AC3: Deduplication & Smart Skipping

1. Do NOT send reminder if:
   - Speaker has already responded (status != INVITED for response reminders)
   - Content already submitted (contentStatus != PENDING for content reminders)
   - Reminder of same tier already sent for same speaker+deadline combination (check `speaker_reminder_log`)
   - Speaker email is null/empty
   - Speaker's `reminders_disabled` flag is set to true
2. Each sent reminder logged in `speaker_reminder_log` table with tier, type, and sent timestamp

### AC4: Reminder Email Templates

1. Six email templates (3 tiers x 2 types):
   - `speaker-reminder-response-tier1-{en,de}.html` - Friendly response reminder
   - `speaker-reminder-response-tier2-{en,de}.html` - Urgent response reminder
   - `speaker-reminder-response-tier3-{en,de}.html` - Final response reminder
   - `speaker-reminder-content-tier1-{en,de}.html` - Friendly content reminder
   - `speaker-reminder-content-tier2-{en,de}.html` - Urgent content reminder
   - `speaker-reminder-content-tier3-{en,de}.html` - Final content reminder
2. Templates include: speaker name, event title, event date, deadline date, days remaining, portal link
3. Tier 3 templates include escalation warning: "If we don't hear from you, we may need to find an alternative speaker"
4. Template follows existing `speaker-invitation-{en,de}.html` HTML structure and branding

### AC5: Reminder Logging & Communication History

1. All reminders logged in new `speaker_reminder_log` database table
2. Each log entry records: speaker_pool_id, reminder_type (RESPONSE/CONTENT), tier (TIER_1/TIER_2/TIER_3), sent_at, email_address
3. Also logged as an `OutreachHistory` entry with `contact_method = 'automated_email'` for unified communication history
4. Organizer can see automated reminders in existing outreach history views

### AC6: Manual Override - Disable Reminders Per Speaker

1. New boolean field `reminders_disabled` on SpeakerPool entity (default: false)
2. When `reminders_disabled = true`, scheduler skips that speaker entirely
3. Organizer can toggle via existing speaker pool PATCH endpoint
4. No new UI needed - organizer uses existing speaker management interface

### AC7: Organizer Escalation Notification

1. After Tier 3 reminder is sent, create an in-app notification for the organizer
2. Notification message: "Speaker {name} has not responded/submitted content despite 3 reminders for {event}. Manual follow-up recommended."
3. Uses existing `NotificationService.createAndSendEmailNotification()` pattern

### AC8: Manual Reminder Trigger by Organizer

1. Organizer can manually send a reminder to a specific speaker via `POST /api/v1/events/{eventCode}/speaker-pool/{speakerPoolId}/send-reminder`
2. Request body specifies `reminderType` (RESPONSE or CONTENT) and optionally `tier` (auto-detected from days remaining if omitted)
3. Reuses the same email templates as the automated scheduler
4. Bypasses the deduplication check (organizer explicitly wants to send)
5. Still respects `reminders_disabled` flag - returns 409 Conflict if disabled
6. Logged in `speaker_reminder_log` with `triggered_by` field set to organizer username (automated reminders have `triggered_by = 'SYSTEM'`)
7. Also logged as `OutreachHistory` entry with `contact_method = 'manual_email'`
8. Requires ORGANIZER role (JWT authenticated, not magic link)

## Dev Notes

### Architecture Patterns & Constraints

**Service**: Event Management Service (`services/event-management-service/`)
- All speaker functionality lives here (same as Stories 6.1-6.4)
- Scheduler job pattern: follow `TokenCleanupScheduler.java` (ShedLock + `@Scheduled`)
- Email template pattern: follow `SpeakerInvitationEmailService.java` (ClassPathResource + variable substitution)
- Communication logging: use existing `OutreachHistory` entity (Story 5.3)

**Scheduling Infrastructure (Already Configured)**:
- `ShedLockConfig.java` - `@EnableScheduling` + `@EnableSchedulerLock(defaultLockAtMostFor = "10m")`
- ShedLock uses PostgreSQL `shedlock` table (V31 migration)
- Pattern: `@Scheduled(cron = "...") @SchedulerLock(name = "...", lockAtLeastFor = "PT5M", lockAtMostFor = "PT30M")`
- Reference: `TokenCleanupScheduler.java` (3 AM daily, same pattern to follow)

**Email Delivery (Already Configured)**:
- Shared-kernel `EmailService.sendHtmlEmail(to, subject, htmlBody)` for AWS SES
- Template variables: `{{variableName}}` syntax, conditional blocks: `{{#var}}...{{/var}}`
- Templates loaded via `ClassPathResource("email-templates/...")` with `StandardCharsets.UTF_8`
- `@Async` for non-blocking email delivery
- Reference: `SpeakerInvitationEmailService.java` (full template loading + variable replacement pattern)

**SpeakerPool Entity Key Fields**:
- `responseDeadline`: `LocalDate` type (NOT Instant or LocalDateTime)
- `contentDeadline`: `LocalDate` type
- `status`: `SpeakerWorkflowState` enum (IDENTIFIED, INVITED, ACCEPTED, DECLINED, CONFIRMED, WITHDREW)
- `contentStatus`: `String` field (PENDING, SUBMITTED, APPROVED, REVISION_NEEDED)
- `email`: `String` - speaker's email address
- `speakerName`: `String` - display name

**Event.date**: `Instant` type (stored as TIMESTAMP in DB)

### Database Schema

```sql
-- V54__Add_speaker_reminder_log.sql

-- Reminder log for deduplication and audit trail
CREATE TABLE speaker_reminder_log (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    speaker_pool_id UUID NOT NULL REFERENCES speaker_pool(id) ON DELETE CASCADE,
    event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
    reminder_type VARCHAR(20) NOT NULL,  -- 'RESPONSE' or 'CONTENT'
    tier VARCHAR(10) NOT NULL,           -- 'TIER_1', 'TIER_2', 'TIER_3'
    email_address VARCHAR(255) NOT NULL,
    deadline_date DATE NOT NULL,
    triggered_by VARCHAR(100) NOT NULL DEFAULT 'SYSTEM',  -- 'SYSTEM' for automated, organizer username for manual
    sent_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
);

-- Deduplication index: prevent sending same automated tier for same speaker+deadline
-- Only applies to SYSTEM-triggered reminders; manual triggers bypass dedup
CREATE UNIQUE INDEX idx_reminder_log_dedup
    ON speaker_reminder_log(speaker_pool_id, reminder_type, tier, deadline_date)
    WHERE triggered_by = 'SYSTEM';

-- Query index: find reminders by speaker
CREATE INDEX idx_reminder_log_speaker
    ON speaker_reminder_log(speaker_pool_id);

-- Query index: find reminders by event
CREATE INDEX idx_reminder_log_event
    ON speaker_reminder_log(event_id);

-- Add reminders_disabled flag to speaker_pool
ALTER TABLE speaker_pool ADD COLUMN IF NOT EXISTS reminders_disabled BOOLEAN DEFAULT FALSE;
```

### Previous Story Intelligence (from 6.4)

**Key learnings to apply:**
1. **EventType enum**: `FULL_DAY`, `AFTERNOON`, `EVENING` (NOT `CONFERENCE`)
2. **EventWorkflowState**: 9-state model in codebase differs from architecture doc's 6-state model. Use actual enum values from shared-kernel.
3. **SpeakerPool.responseDeadline/contentDeadline**: `LocalDate` type (compare with `LocalDate.now()`)
4. **Event.date**: `Instant` type - use `Instant.now()` for comparison
5. **TestUserApiClientConfig**: Provides mock `UserApiClient` automatically in tests
6. **Integration test setup**: Clean repos in FK order, use `System.currentTimeMillis()` for unique IDs
7. **SecurityConfig dual update**: Needed for the manual trigger endpoint (AC8) - add to BOTH api-gateway and event-management-service SecurityConfig (requires ORGANIZER role, NOT permitAll)

### Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Email sending failures | Catch exceptions per speaker, log failure, continue with remaining speakers |
| ShedLock contention | Use appropriate lock durations (min 5m, max 30m) |
| Large number of speakers | Batch processing with configurable batch size |
| Timezone issues | Use `LocalDate.now()` for deadline comparison (consistent with SpeakerPool field type) |
| Template rendering errors | Wrap in try-catch, log error, skip that speaker's reminder |
