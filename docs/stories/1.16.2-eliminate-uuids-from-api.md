# Story 1.16.2: Eliminate UUIDs from API - Use Only Meaningful IDs

## Status
Accepted

## Story

**As a** developer and API consumer,
**I want** the API to expose only meaningful, human-readable IDs (eventCode, username, company name),
**so that** the API is cleaner, more intuitive, and UUIDs are kept as internal database implementation details only.

## Domain Context

### Primary Domain
**Cross-cutting Infrastructure** - Affects all bounded contexts:
- Event Management (eventCode)
- Company Management (company name)
- User Management (username)
- All domain events (remove ID wrappers)

### Involved Services
- **Shared Kernel:** Remove ID wrapper types, update DomainEvent base class
- **Event Management Service:** Use eventCode in API, DTO projections
- **Company-User Management Service:** Use username and company name in API, DTO projections

### Architectural Decision

**Database Layer (Internal):**
- UUID primary keys remain (performance, stability, immutability)
- Foreign keys use UUIDs internally
- Meaningful IDs are unique alternate keys (indexed)

**API Layer (Public):**
- Only meaningful IDs exposed in requests/responses
- No UUIDs in DTOs, OpenAPI specs, or domain events
- DTO projections map UUID FKs → meaningful IDs

**Strategy:**
```
Database: events.id (UUID PK), events.event_code (unique alternate key)
API:      GET /events/BATbern56 → { "id": "BATbern56", "organizerId": "john.doe" }
```

## Requirements Context

### Related Functional Requirements
- **FR-API-001:** APIs must use human-readable identifiers
- **FR-DX-001:** Developer experience - clean, intuitive API contracts
- **FR-PERF-001:** Database performance - UUID PKs for joins/indexes

### Architecture Alignment
- **Prerequisite:** Story 1.16.1 (Meaningful IDs infrastructure) must be completed first
- **Builds on:** Existing dual-identifier pattern (UUID PK + meaningful alternate key)
- **Removes:** ID wrapper complexity (EventId, UserId, CompanyId, SpeakerId)

## Acceptance Criteria

### Shared Kernel (AC 1-3)

1. **AC1:** ID wrapper classes are completely removed
   - EventId.java, UserId.java, CompanyId.java, SpeakerId.java deleted
   - All references to these types removed from codebase
   - No compilation errors after deletion

2. **AC2:** DomainEvent base class uses String for aggregate IDs
   - `DomainEvent<UUID>` changed to `DomainEvent<String>`
   - `aggregateId` field holds meaningful ID (eventCode, username, company name)
   - `userId` field is plain String (not UserId wrapper)

3. **AC3:** SlugGenerationService provides username generation
   - `generateUsername(firstName, lastName)` method implemented
   - Handles German characters (ä→ae, ö→oe, ü→ue)
   - `ensureUniqueUsername()` handles collisions (.2, .3 suffixes)

### Event Management Service (AC 4-7)

4. **AC4:** Event DTOs use only meaningful IDs (no UUIDs)
   - EventResponse.id returns eventCode (e.g., "BATbern56")
   - EventResponse.organizerId returns username (e.g., "john.doe")
   - No UUID fields in any DTO

5. **AC5:** Event API endpoints use eventCode in paths
   - `GET /api/v1/events/{eventCode}` accepts eventCode
   - `PUT /api/v1/events/{eventCode}` uses eventCode
   - `DELETE /api/v1/events/{eventCode}` uses eventCode

6. **AC6:** Event domain events use String IDs (no wrappers)
   - EventCreatedEvent extends `DomainEvent<String>`
   - aggregateId = eventCode
   - userId = username (plain String)

7. **AC7:** DTO projections used for all read queries
   - Repository projection queries join to resolve FKs
   - Single query returns complete DTO (no N+1)
   - Projections use constructor expressions in JPQL

### Company-User Management Service (AC 8-12)

8. **AC8:** User entity has username field with generation
   - Database migration adds username column (unique, indexed)
   - Username auto-generated from firstName.lastName
   - Collision handling appends .2, .3, etc.

9. **AC9:** User DTOs use only meaningful IDs
   - UserResponse.id returns username
   - UserResponse.companyId returns company.name
   - No UUID fields in any user DTO

10. **AC10:** Company DTOs use name as id
    - CompanyResponse.id returns company.name (e.g., "GoogleZH")
    - displayName field shows full name (e.g., "Google Zürich AG")
    - No UUID id field

11. **AC11:** User/Company API endpoints use meaningful IDs
    - `GET /api/v1/users/{username}` accepts username
    - `GET /api/v1/companies/{companyName}` accepts company name (case-insensitive)
    - All endpoints return DTOs with meaningful IDs only

12. **AC12:** Company/User domain events use String IDs
    - CompanyCreatedEvent aggregateId = company.name
    - All events use plain String userId (no wrapper)

### OpenAPI & Testing (AC 13-14)

13. **AC13:** OpenAPI specs contain no UUID types
    - All `format: uuid` removed from specs
    - String patterns define formats (e.g., `^BATbern[0-9]+$`)
    - Examples use meaningful IDs

14. **AC14:** All tests use meaningful IDs
    - Test data builders use eventCode, username, company.name
    - Integration tests assert on meaningful IDs
    - No UUID assertions in response validation

## Dev Notes - Implementation Guide

### Key Architectural Decisions

**1. Keep UUIDs in Database**
- Primary keys remain UUID (performance, immutability)
- Foreign keys use UUID internally
- Meaningful IDs are unique alternate keys

**2. DTO Projections for All Reads**
- Use JPQL constructor expressions
- JOIN to resolve FK UUIDs → meaningful IDs
- Single query returns complete DTO (no N+1)

**3. Service Layer Resolves IDs**
- Requests provide meaningful IDs (username, company.name)
- Service resolves to entities (uses UUID internally)
- Responses use DTO projections (meaningful IDs only)

### DTO Projection Pattern

```java
// DTO Record
public record EventResponseDto(
    String id,          // eventCode
    String organizerId, // username
    String title
    // ... no UUIDs
) {}

// Repository Projection
@Query("""
    SELECT new ch.batbern.events.dto.EventResponseDto(
        e.eventCode,
        u.username,
        e.title
    )
    FROM Event e
    JOIN e.organizer u
    WHERE e.eventCode = :code
""")
Optional<EventResponseDto> findEventDtoByCode(@Param("code") String eventCode);

// Service Usage
public EventResponseDto getEvent(String eventCode) {
    return eventRepository.findEventDtoByCode(eventCode)
        .orElseThrow(() -> new EventNotFoundException(eventCode));
}
```

### Meaningful ID Resolution Pattern

```java
// Create flow: meaningful ID → entity
public EventResponseDto createEvent(CreateEventRequest request) {
    // Resolve username → User entity
    User organizer = userRepository.findByUsername(request.getOrganizerId())
        .orElseThrow(() -> new UserNotFoundException(request.getOrganizerId()));

    // Create entity with UUID FKs
    Event event = Event.builder()
        .eventCode("BATbern" + request.getEventNumber())
        .organizer(organizer)  // Uses UUID FK internally
        .build();

    Event saved = eventRepository.save(event);

    // Return DTO with meaningful IDs
    return eventRepository.findEventDtoByCode(saved.getEventCode())
        .orElseThrow();
}
```

### Username Generation Pattern

```java
// Auto-generate with collision handling
String baseUsername = slugService.generateUsername(
    request.getFirstName(),
    request.getLastName()
);

String username = slugService.ensureUniqueUsername(
    baseUsername,
    userRepository::existsByUsername
);

// Result: john.doe, or john.doe.2 if collision
```
