# Story 4.1.5c: Secure Email-Based Registration Confirmation

## Dependencies
**Depends on**: Story 4.1.5 (Registration Wizard), Story 4.1.6 (Registration Confirmation)

## Status
Done

## Story
**As a** security-conscious product owner,
**I want** to remove the QR code system and implement secure email-based confirmation,
**so that** registration data is not exposed in URLs or accessible to unauthorized users.

## Background

### Security Issues Identified
1. **Registration codes in URLs** - Exposed in browser history, screenshots, server logs
2. **Public QR endpoint** - Anyone with registration code could access QR codes
3. **No authentication** - Confirmation codes acted as bearer tokens without expiry
4. **Screenshot vulnerability** - QR codes could be copied and reused

### Decision
Remove QR code system entirely. Event check-in will be handled via:
- Manual registration list check at event desk
- Email confirmation as proof of registration
- Future: Account-based registration management via Cognito

## Domain Context

### Primary Domain
Attendee Experience Domain (Registration confirmation)

### Involved Services
- web-frontend (React 19 + TypeScript)
- Event Management Service (via API Gateway)
- API Gateway: Registration and confirmation endpoints

### Cross-Domain Dependencies
- Email service sends confirmation emails with JWT tokens
- Account linking via ADR-005 (when user creates Cognito account)

### Security Improvements

| Aspect | Before | After |
|--------|--------|-------|
| **URL Exposure** | Registration code in URL | Clean URL, no sensitive data |
| **Access Control** | Public QR endpoint | JWT token required (48h expiry) |
| **Attack Vector** | Screenshot sharing | Email-only access |
| **Persistence** | Permanent access | Time-limited (48h) |
| **Browser History** | Registration codes logged | No sensitive data |
| **Validation** | None (bearer token) | JWT signature + expiry check |

**Threat Model:**

- ❌ Screenshot attacks → ✅ Mitigated (no QR codes)
- ❌ Permanent access → ✅ Mitigated (48h token expiry)

**Recommendations:**
1. Add rate limiting to confirmation endpoint (10 req/min per IP)

3. Consider token revocation on successful confirmation (one-time use)
