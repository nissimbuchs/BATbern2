# Story 1.15a.1b: Session-User Many-to-Many Relationship (Speaker Assignments)

## Status
✅ **COMPLETE** - All 17 tasks finished (100%)

## Story
**As a** backend developer,
**I want** sessions to support multiple speakers via a many-to-many relationship with users,
**so that** the frontend can display speaker information for Story 4.1.4 without cross-service dependencies.

**ARCHITECTURAL NOTE**: This implements ADR-004 pattern where domain entities reference User directly via userId FK, reducing dependency on speaker-coordination-service.

## Domain Context

### Primary Domain
Event Management Domain

### Involved Services
- Event Management Service (Java/Spring Boot)
- Company User Management Service (User entity via userId FK)

### Cross-Domain Dependencies
- References User entity via userId (ADR-004 pattern)
- Eliminates dependency on speaker-coordination-service for basic speaker display

## Requirements Context

### Related Functional Requirements
- Epic 1, Story 1.15a.1 (Events API Consolidation)
- Story 4.1.4 (Homepage Event Content Sections - requires speaker data)
- ADR-004 (Factor User Fields from Domain Entities)

### Architecture Impact
- Implements `session_users` junction table for many-to-many relationship
- Sessions reference Users directly (not Speakers) for reduced coupling
- Speaker-coordination-service can add workflow-specific data independently later

## Dev Notes - Implementation Guide

### ADR-004 Reference Pattern

**Key Principle**: Domain entities reference User via `userId` UUID FK, NOT via @ManyToOne relationship.

```java
// ✅ CORRECT - SessionUser references User via userId
@Entity
public class SessionUser {
    @Column(name = "user_id", nullable = false, columnDefinition = "UUID")
    private UUID userId;  // FK to user_profiles.id (different database)

    // NOT a @ManyToOne because User is in a different database
}

// ❌ WRONG - Would require cross-database join
@Entity
public class SessionUser {
    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;  // Can't join across databases
}
```

### Cross-Service User Data Lookup

**Option A**: Direct database query (if shared database access)
```java
@Query("""
    SELECT new SessionSpeakerResponse(
        u.username, u.firstName, u.lastName, c.name, u.profilePictureUrl,
        su.speakerRole, su.presentationTitle, su.isConfirmed
    )
    FROM SessionUser su, User u
    WHERE su.userId = u.id AND su.session.id = :sessionId
""")
List<SessionSpeakerResponse> findSessionSpeakersWithUserData(@Param("sessionId") UUID sessionId);
```

**Option B**: REST API call to User Service
```java
public SessionSpeakerResponse enrichWithUserData(SessionUser sessionUser) {
    User user = userServiceClient.getUserById(sessionUser.getUserId());
    return SessionSpeakerResponse.builder()
        .username(user.getUsername())
        .firstName(user.getFirstName())
        // ... map other fields
        .speakerRole(sessionUser.getSpeakerRole())
        .build();
}
```

**Recommendation**: Use Option A (direct query) if both services share the same PostgreSQL database, which is the case for BATbern.

### Database Migration Notes

**Important**: The `user_id` column is a logical foreign key (documented in comments) but NOT enforced with a database-level FOREIGN KEY constraint because user_profiles table may be in a different database schema.

```sql
-- Logical FK documented in comment, not enforced by database
user_id UUID NOT NULL, -- FK to user_profiles.id in company-user-management-service
```

If both services share the same database, you CAN add the constraint:
```sql
user_id UUID NOT NULL REFERENCES user_profiles(id) ON DELETE CASCADE,
```
