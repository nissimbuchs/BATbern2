# Story 1.2.6: Migrate to Database-Only Roles Architecture

## Status

Done

**Completion Date:** 2025-10-22
**Implementation Summary:** Successfully migrated from `cognito:groups` (array) to `custom:roles` (comma-separated string) across all services, aligned with ADR-001 database-centric architecture.

## Story

**As a** platform architect,
**I want** to migrate existing code from Cognito Groups to database-only roles,
**so that** the implementation aligns with ADR-001's database-centric architecture and eliminates complexity from bidirectional sync.

**Context**: Per ADR-001, BATbern has adopted a database-centric architecture where roles are stored exclusively in PostgreSQL and synced to JWT via PreTokenGeneration Lambda. However, existing code in Story 1.14-2 still references `cognito:groups` and contains stub implementations for Cognito sync. This story removes all Cognito Groups references and updates code to use `custom:roles` JWT claim.

**Dependencies:**
- **DEPENDS ON: Story 1.14-2 (User Management Service Foundation)** - Implemented `user_profiles` and `role_assignments` tables
- **DEPENDS ON: Story 1.2.5 (User Sync Implementation)** - Implements PostConfirmation and PreTokenGeneration Lambdas
- **BLOCKS: Story 1.2.3 (Account Creation Flow)** - Registration flow depends on correct JWT claim extraction

## Acceptance Criteria

1. **Spring Security extracts roles from custom:roles claim**
   - When JWT contains `custom:roles: "ATTENDEE,SPEAKER"`
   - Then Spring Security grants `ROLE_ATTENDEE` and `ROLE_SPEAKER` authorities
   - And `@PreAuthorize("hasRole('SPEAKER')")` annotations work correctly

2. **API Gateway UserContextExtractor reads custom:roles**
   - When JWT is decoded in API Gateway
   - Then `UserContext.role` contains primary role (first in list)
   - And `UserContext.additionalRoles` contains remaining roles
   - And no errors occur for missing `cognito:groups` claim

3. **Frontend auth service handles multiple roles**
   - When JWT is decoded in frontend
   - Then `UserContext.role` contains primary role
   - And `UserContext.roles` array contains all roles
   - And role checks work correctly

4. **No Cognito Groups operations in code**
   - When searching codebase for `cognito:groups`
   - Then only ADR-001 and documentation references exist
   - And no active code reads or writes to Cognito Groups
   - And no `adminAddUserToGroup` or `adminRemoveUserFromGroup` calls exist

5. **All existing tests pass with custom:roles**
   - When running test suite
   - Then SecurityConfig tests use `custom:roles` fixtures
   - And UserContextExtractor tests pass with updated JWT mocks
   - And integration tests pass end-to-end

6. **Backward compatibility handled gracefully**
   - When JWT contains `cognito:groups` but not `custom:roles` (legacy token)
   - Then fallback to empty roles (user has no access, must re-login)
   - And no exceptions thrown

  private static class CustomRolesToAuthoritiesConverter implements Converter<Jwt, Collection<GrantedAuthority>> {
      @Override
      public Collection<GrantedAuthority> convert(Jwt jwt) {
          // Extract roles from custom:roles claim (comma-separated string)
          String rolesString = jwt.getClaimAsString("custom:roles");

          if (rolesString == null || rolesString.isEmpty()) {
              return Collections.emptyList();
          }

          // Split comma-separated roles and map to ROLE_ authorities
          return Arrays.stream(rolesString.split(","))
              .map(role -> new SimpleGrantedAuthority("ROLE_" + role.trim().toUpperCase()))
              .collect(Collectors.toList());
      }
  }

## Dev Notes - Implementation Guide

### Migration Strategy

**IMPORTANT**: This migration requires coordination with Story 1.2.5 deployment.

**Deployment Order**:
1. **Deploy Story 1.2.5** - PreTokenGeneration Lambda that adds `custom:roles` claim
2. **Wait for all users to get new tokens** - Tokens have 1-hour TTL, so wait at least 1 hour
3. **Deploy Story 1.2.6** - Backend code reading `custom:roles`
4. **Verify** - Check logs for any auth errors

**Backward Compatibility**:
- For 1 hour after Lambda deployment, some users will have old tokens with `cognito:groups`
- Code handles this by returning empty roles (users must re-login)
- No hard failures, graceful degradation

### Code Changes Summary

**What Changes**:
1. JWT claim name: `cognito:groups` → `custom:roles`
2. Claim format: Array of strings → Comma-separated string
3. Multiple roles: Supported (e.g., "ATTENDEE,SPEAKER")

**What Stays the Same**:
1. Authority prefix: `ROLE_` (Spring Security convention)
2. Role names: ORGANIZER, SPEAKER, PARTNER, ATTENDEE (uppercase in JWT)
3. `@PreAuthorize` annotations in controllers

### Testing Strategy

**Unit Tests**:
- Test claim extraction with various formats
- Test graceful handling of missing claims
- Test authority mapping

**Integration Tests**:
- Test end-to-end with real JWT tokens from Cognito + PreTokenGeneration Lambda
- Verify `@PreAuthorize` annotations work
- Test multiple roles scenario

**Manual Testing**:
1. Register new user (Story 1.2.3)
2. Verify JWT contains `custom:roles: "ATTENDEE"`
3. Promote user to SPEAKER via admin API
4. Re-login, verify JWT contains `custom:roles: "ATTENDEE,SPEAKER"`
5. Test SPEAKER-only endpoint access
