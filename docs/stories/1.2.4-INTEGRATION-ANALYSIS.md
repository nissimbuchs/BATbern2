## ⚠️ CRITICAL GAPS: What Needs Verification

### ❌ GAP 1: Backend API Strategy Unclear

**Issue**: Story proposes creating backend endpoints `/api/v1/auth/verify-email` and `/api/v1/auth/resend-verification` (lines 348-371, 374-413).

**BUT**: Cognito email verification is handled client-side with Amplify!

**Codebase Finding**:
- `web-frontend/src/services/auth/authService.ts` uses Amplify v6
- Amplify already has `confirmSignUp` and `resendSignUpCode` functions
- Story 1.2.3 uses direct Cognito integration (NO backend proxy)

**Correct Pattern**:
```typescript
import { confirmSignUp, resendSignUpCode } from 'aws-amplify/auth';

// Email verification (client-side)
await confirmSignUp({ username: email, confirmationCode: code });

// Resend code (client-side)
await resendSignUpCode({ username: email });
```

**Decision Needed**:
- ❌ Do NOT create backend `/api/v1/auth/verify-email` endpoint (frontend calls Cognito directly)
- ❌ Do NOT create `EmailVerificationService.java` (unnecessary proxy)
- ✅ Use Amplify v6 `confirmSignUp` and `resendSignUpCode` directly

**Files to SKIP**:
- `api-gateway/src/main/java/ch/batbern/api/auth/service/EmailVerificationService.java`
- `api-gateway/src/main/java/ch/batbern/api/auth/dto/EmailVerificationRequest.java`
- `api-gateway/src/main/java/ch/batbern/api/auth/dto/EmailVerificationResponse.java`

**Files to CREATE**:
- `web-frontend/src/services/auth/emailVerificationService.ts` (Amplify wrapper)

### ❌ GAP 2: Cognito Email Configuration Not Mentioned

**Issue**: Story assumes AWS SES custom templates (lines 176-180, 276-335), but Cognito has BUILT-IN email verification templates!

**Actual AWS Cognito Behavior**:
- Cognito User Pools have default verification email templates
- Custom templates configured in Cognito User Pool settings (NOT AWS SES)
- Template customization: AWS Console → Cognito → User Pool → Message customization

**Decision Needed**:
- ❌ Do NOT create AWS SES templates for email verification (Cognito handles it)
- ✅ Configure Cognito User Pool email templates (in infrastructure or AWS Console)
- ✅ Cognito auto-sends verification email after signUp (if email verification required)

**Correct Approach**:
1. Verify Cognito User Pool has email verification enabled
2. Customize Cognito email templates (in User Pool settings)
3. Frontend calls `resendSignUpCode` to trigger resend (Cognito sends email)

**Files to SKIP**:
- AWS SES template `EmailVerificationDE`
- AWS SES template `EmailVerificationEN`
- Task 1: AWS SES Email Template Setup (lines 175-180)

**Infrastructure Check Needed**:
- Check `infrastructure/lib/stacks/cognito-stack.ts` for User Pool email settings
- Verify email verification is required
- Check if custom email templates configured

### ⚠️ GAP 7: Role-Based Redirect Logic Incomplete

**Issue**: Story mentions role-based redirect (lines 249-254, 601) but doesn't explain:
- HOW to get user role after verification?
- WHERE role-based routing logic lives?
- WHAT if user has no role yet? (PostConfirmation Lambda assigns ATTENDEE role in Story 1.2.5)

**Story Says** (line 601):
```typescript
navigate('/attendee/dashboard'); // Default to ATTENDEE dashboard
```

**But WHEN does user get ATTENDEE role?**
- Story 1.2.5: PostConfirmation Lambda creates user_profiles and role_assignments
- PostConfirmation Lambda triggers AFTER `confirmSignUp` succeeds
- User should have role immediately after verification

**Correct Flow**:
1. `confirmSignUp()` succeeds
2. Cognito triggers PostConfirmation Lambda (Story 1.2.5)
3. Lambda assigns ATTENDEE role to Cognito group
4. Frontend: Fetch updated auth session to get role
5. Navigate based on role

**Missing Code**:
```typescript
// After confirmSignUp succeeds
await confirmSignUp({ username: email, confirmationCode: code });

// NEED TO: Refresh auth session to get updated role
const session = await fetchAuthSession({ forceRefresh: true });
const role = session.tokens?.idToken?.payload['cognito:groups']?.[0];

// Navigate based on role
navigate(`/${role.toLowerCase()}/dashboard`);  // /attendee/dashboard
```

**Decision Needed**:
- Add `fetchAuthSession({ forceRefresh: true })` after verification
- Extract role from token
- Use role-based navigation (not hardcoded `/attendee/dashboard`)

### ⚠️ GAP 8: Email Link Format Not Aligned with Frontend

**Issue**: Story specifies email link format (line 62):
```
/auth/verify-email?code={code}&email={email}
```

But doesn't explain:
- WHO generates this link? (Cognito? Backend? Frontend?)
- WHERE is this link configured? (Cognito User Pool settings?)

**Actual Cognito Behavior**:
- Cognito sends verification CODE (6 digits) in email
- Cognito does NOT automatically generate clickable verification links
- Custom verification link requires:
  1. Cognito email template customization (add link with `{####}` placeholder)
  2. Frontend URL in Cognito template: `https://yourdomain.com/auth/verify-email?code={####}&email={email}`

**Decision Needed**:
- Update Cognito User Pool email template to include verification link
- Link format: `${FRONTEND_URL}/auth/verify-email?code={####}&email=${email}`
- `{####}` is Cognito placeholder for verification code

**Infrastructure Task**:

