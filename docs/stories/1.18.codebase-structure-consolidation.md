# Story 1.18: Codebase Structure Consolidation

## Status
Draft

## Story

**As a** developer,
**I want** the codebase structure to be organized, consistent, and aligned with maintainability best practices,
**so that** the project is easier to navigate, understand, and maintain as it grows.

## Domain Context

### Primary Domain
Infrastructure & Development Operations - Codebase organization and structural improvements

### Scope
This is a **REFACTORING STORY** focused on improving codebase organization without changing functionality:
- Service directory naming standardization
- Test consolidation into unified structure
- Infrastructure stack organization
- Documentation alignment with actual structure

### What This Story Does NOT Include
- **Excluded**: Deleting or moving `BATspa-old/` (legacy Angular app - keep as-is)
- **Excluded**: Deleting or moving `apps/projectdoc/` (project documentation - keep as-is)
- **Excluded**: Functionality changes or feature additions
- **Excluded**: Breaking changes to deployed systems

## Requirements Context

### Problem Statement
Current codebase structure analysis (2025-11-01) revealed:
1. **Service Naming Inconsistency**: All 5 services have `-service` suffix not present in documentation
2. **Test Fragmentation**: E2E tests scattered across 3 locations (`bruno-tests/`, `web-frontend/e2e/`, `infrastructure/test/e2e/`)
3. **Infrastructure Organization**: All CDK stacks flat in single directory, not organized by concern (global/core/domain)
4. **Documentation Misalignment**: `source-tree.md` doesn't match actual structure

### Goals
- **Maintainability**: Reduce cognitive load for developers navigating the codebase
- **Consistency**: Align service naming with documentation standards
- **Organization**: Group related code together (tests, stacks)
- **Clarity**: Make project structure self-documenting

### Related Documentation
- **Source Tree**: `docs/architecture/source-tree.md` (needs update)
- **Analysis Report**: Comprehensive structure analysis completed 2025-11-01 (see Dev Notes)

## Architecture Context

### Current Structure Issues

**Services Directory:**
```
services/
├── attendee-experience-service/        # Should be: attendee-experience/
├── company-user-management-service/    # Should be: company-management/
├── event-management-service/           # Should be: event-management/
├── partner-coordination-service/       # Should be: partner-coordination/
└── speaker-coordination-service/       # Should be: speaker-coordination/
```

**Test Fragmentation:**
- `bruno-tests/` - API integration tests (not documented)
- `web-frontend/e2e/` - Frontend e2e tests
- `infrastructure/test/e2e/` - Infrastructure tests
- Documentation expects: unified `e2e-tests/` directory

**Infrastructure Organization:**
- Current: All stacks in flat `infrastructure/lib/stacks/` (19 files)
- Target: Organized by concern (`global/`, `core/`, `domain/`, per documentation)

### Target Structure (Meet in Middle Approach)

**Services (Remove `-service` suffix):**
```
services/
├── attendee-experience/
├── company-management/
├── event-management/
├── partner-coordination/
└── speaker-coordination/
```

**Tests (Consolidated):**
```
e2e-tests/
├── api/                    # From bruno-tests/
├── workflows/              # From web-frontend/e2e/
├── infrastructure/         # From infrastructure/test/e2e/
├── fixtures/               # Shared test data
└── README.md              # Test documentation
```

**Infrastructure (Organized by Concern):**
```
infrastructure/lib/stacks/
├── global/                 # DNS, Secrets, CI/CD
├── core/                   # Network, Database, Cognito, EventBus
├── domain/                 # Service-specific stacks
└── frontend/               # Frontend deployment
```

### Architecture Patterns
- **DDD Alignment**: Service names match bounded contexts (no technical suffixes)
- **Test Organization**: Tests organized by type (api, workflows, infrastructure)
- **Infrastructure as Code**: CDK stacks organized by deployment lifecycle
- **Documentation Accuracy**: source-tree.md reflects reality

- Development-only environment (no live deployments)

4. **CI/CD**: Update GitHub Actions workflows alongside code changes
