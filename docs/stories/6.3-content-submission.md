## Story

**As a** speaker,
**I want** to submit my presentation content (title, abstract) and upload materials through a self-service portal,
**so that** organizers don't have to manually collect and upload my content.

```sql
CREATE TABLE IF NOT EXISTS speaker_content_submissions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    speaker_pool_id UUID NOT NULL REFERENCES speaker_pool(id) ON DELETE CASCADE,
    session_id UUID REFERENCES sessions(id),
    title VARCHAR(200) NOT NULL,
    abstract TEXT NOT NULL,
    abstract_char_count INTEGER NOT NULL,
    submission_version INTEGER DEFAULT 1,
    reviewer_feedback TEXT,
    submitted_at TIMESTAMP DEFAULT NOW(),
    reviewed_at TIMESTAMP,
    reviewed_by VARCHAR(255),
    CONSTRAINT chk_abstract_length CHECK (abstract_char_count <= 1000)
);

CREATE INDEX IF NOT EXISTS idx_content_submissions_speaker ON speaker_content_submissions(speaker_pool_id);
CREATE INDEX IF NOT EXISTS idx_content_submissions_session ON speaker_content_submissions(session_id);
```

## Acceptance Criteria

### AC1: Session Assignment Check
1. Content submission page validates speaker has assigned session
2. If no session assigned, show message: "Contact organizer to assign your session"
3. Block submission until `session_id` is set on SpeakerPool
4. API returns 400 error if submit attempted without session

### AC2: Title Input
1. Title field with 200 character max
2. Real-time character counter showing `{current}/200`
3. Required field validation
4. Clear, descriptive title guidance tooltip

### AC3: Abstract Input
1. Abstract textarea with 1000 character max
2. Real-time character counter showing `{current}/1000`
3. Quality hints displayed:
   - "Include what attendees will learn"
   - "Add key takeaways"
   - "Must include 'Lessons Learned' from real experience"
4. Warning if abstract < 200 characters

### AC4: Draft Auto-Save
1. Auto-save every 30 seconds to prevent data loss
2. Visual indicator: "Auto-saved 30 seconds ago"
3. Manual "Save Draft" button available
4. Resume from draft on page reload
5. Draft stored server-side (not just localStorage)

### AC5: Content Submission
1. Submit button validates all required fields
2. On submit: `content_status` → `SUBMITTED`
3. Creates `ContentSubmission` record with version 1
4. Sets `content_submitted_at` timestamp
5. Success page shows:
   - Confirmation message
   - What happens next (quality review)
   - Organizer contact info

### AC6: Organizer Notification
1. `SpeakerContentSubmittedEvent` published on submission
2. Organizer receives notification (dashboard + optional email)
3. Content appears in quality review queue

### AC11: Automatic Status History Tracking

1. **Status History**: When speaker submits content, automatically creates `SpeakerStatusHistory` record
   - `previous_status`: ACCEPTED (or current status)
   - `new_status`: CONTENT_SUBMITTED
   - `change_reason`: "Content submitted via speaker portal (version {version})"
   - `changed_by_username`: Speaker's username or speaker name
2. History includes `session_id` and `event_id` for context
3. Only creates history when status actually changes (ACCEPTED → CONTENT_SUBMITTED)
4. Provides audit trail for content submission workflow

### AC7: File Upload (Optional)
1. Presentation upload area with drag-and-drop
2. Accepted formats: PPTX, PDF, KEY
3. Maximum file size: 50MB
4. Upload uses presigned URL pattern (ADR-002)
5. Progress indicator during upload
6. File can be uploaded anytime after acceptance (not required for content approval)

### AC8: Revision Support
1. If `content_status` = `REVISION_NEEDED`, show reviewer feedback
2. Speaker can edit and resubmit
3. New submission increments `submission_version`
4. Previous versions preserved in history

### AC9: Hybrid Workflow
1. Organizer can still submit content on speaker's behalf (existing endpoint)
2. Both paths update same `content_status`
3. No conflict when both paths used

### AC10: Speaker Portal Navigation
1. Profile page shows "Submit Content" link when speaker has session assigned
2. Content submission page shows "Edit Profile" link
3. Both links preserve the VIEW token in URL query parameter
4. Navigation uses same token for seamless experience (no re-authentication)
5. Links are visually prominent and accessible (keyboard navigable)

**Rationale**: Architecture specifies flow "Edit profile → Submit content" (epic-6-speaker-onboarding-plan.md:1108). Without navigation, speakers cannot discover or access content submission page.

## Dev Notes - Implementation Guide

### API Contracts

```yaml
# GET /api/v1/speaker-portal/content?token={token}
Response: 200 OK
  title: string
  abstract: string
  status: "PENDING" | "SUBMITTED" | "APPROVED" | "REVISION_NEEDED"
  reviewerFeedback?: string
  submittedAt?: timestamp
  version?: number
  sessionTitle?: string

# POST /api/v1/speaker-portal/content/draft
Request:
  token: string
  title: string (max 200)
  abstract: string (max 1000)
Response: 200 OK
  draftId: string
  savedAt: timestamp

# POST /api/v1/speaker-portal/content/submit
Request:
  token: string
  title: string (required)
  abstract: string (required)
Response: 201 Created
  submissionId: string
  version: number
  status: "SUBMITTED"
  sessionTitle: string
Error 400: "Session not assigned - contact organizer"
Error 400: "Title is required"
Error 400: "Abstract is required"

# POST /api/v1/speaker-portal/materials/presigned-url
Request:
  token: string
  fileName: string
  contentType: string (application/pdf, application/vnd.ms-powerpoint, etc.)
Response: 200 OK
  uploadUrl: string
  uploadId: string
  expiresIn: number (900 seconds)

# POST /api/v1/speaker-portal/materials/confirm
Request:
  token: string
  uploadId: string
  materialType: "PRESENTATION" | "DOCUMENT"
Response: 200 OK
  materialId: string
  fileName: string
  cloudfrontUrl: string
```

### Technical Design Notes

**Key Patterns:**
- Reuse MagicLinkService for token validation (from 6.1a)
- Reuse PresignedUrlService for file uploads (from 6.2b)
- Follow ContentStatus enum from architecture plan
- Use @Transactional for submission to ensure atomic update

**Content Status Flow:**
```
PENDING → (speaker submits) → SUBMITTED → (organizer reviews) → APPROVED
                                    ↓
                             REVISION_NEEDED → (speaker fixes) → SUBMITTED
```

**Domain Event:**
```java
public record SpeakerContentSubmittedEvent(
    String speakerPoolId,
    String username,
    String eventCode,
    String sessionId,
    String submissionId,
    int submissionVersion,
    Instant submittedAt
) implements DomainEvent {}
```
