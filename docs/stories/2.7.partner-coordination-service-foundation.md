# Story 2.7: Partner Coordination Service Foundation

## Status
Ready for Review

## Story

**As a** partner company representative,
**I want** the foundational Partner Coordination Service with CRUD operations through RESTful APIs,
**so that** I can manage partnership relationships, contacts, topic votes, and meeting attendance efficiently.

## Domain Context

### Primary Domain
Partner Coordination Domain - Partnership lifecycle management, topic voting, meeting coordination

### Involved Services
- **Partner Coordination Service** (primary - `services/partner-coordination-service/`)
- **API Gateway** (routing, authentication middleware)
- **Shared Kernel** (exception hierarchy, domain events, query utilities)
- **OpenAPI Code Generation** (generate API interfaces from partners-api.openapi.yml)
- **Company Management Service** (HTTP API calls for company data)
- **User Management Service** (HTTP API calls for user data)

**Shared-Kernel Foundation** (automatically included):
- Exception hierarchy (NotFoundException, ValidationException, BusinessException)
- ErrorResponse with correlationId, path, severity
- DomainEvent & DomainEventPublisher
- Query utilities (FilterParser, SortParser, PaginationUtils, FieldSelector, IncludeParser)

### Cross-Domain Dependencies
- **Depends on:** Story 2.1 (Company Management Service) - Partner stores `companyName` (meaningful ID via HTTP API)
- **Depends on:** Story 2.1b (User Management Service) - PartnerContact stores `username` (meaningful ID via HTTP API)
- **Depends on:** Story 1.15a (API Consolidation Foundation) - Uses FilterParser, SortParser utilities
- **Integration:** EventBridge for domain events (PartnerCreatedEvent, PartnerContactAddedEvent)

- NO direct database access to Company/User tables
- NO JPQL joins across service boundaries
- Store meaningful IDs only: `companyName`, `username`
- HTTP calls to `GET /api/v1/companies/{companyName}` and `GET /api/v1/users/{username}` for full data

## Architecture Context

### Architecture Patterns

**Data Models** (03-data-architecture.md, lines 1676-1754):

- **TopicSuggestion** (partner-submitted topic ideas)

- RESTful endpoints: `/api/v1/partners`

- PartnerContact enrichment via HTTP calls to User API (NOT JPQL joins)

**Backend Patterns** (06-backend-architecture.md):
- Spring Boot 3.2 + Java 21
- DDD layered architecture (Controller → Service → Repository)
- JPA entities with PostgreSQL
- Domain events to EventBridge
- OpenAPI code generation (importMappings for shared-kernel types)
- **HTTP clients for inter-service communication (RestTemplate or WebClient)**

**Microservices Data Access (ADR-003):**
- Store meaningful IDs: `companyName`, `username` (NOT UUIDs)
- Call Company API: `GET /api/v1/companies/{companyName}` for full company data
- Call User API: `GET /api/v1/users/{username}` for full user data
- NO cross-service JPQL joins (each service owns its data)

## Acceptance Criteria

Following Epic 2 CRUD pattern (Stories 2.1-2.3) with microservices architecture (ADR-003):

1. **Partner aggregate with DDD patterns**
   - Partner entity with JPA annotations
   - Partnership tiers: BRONZE, SILVER, GOLD, PLATINUM, STRATEGIC
   - Active status tracking (start/end dates)
   - Stores `companyName` (String) - meaningful ID per ADR-003
   - **NO UUID foreign key to companies table**

2. **REST API implementing basic CRUD**
   - `GET /api/v1/partners` - List all partners with filtering
   - `GET /api/v1/partners/{companyName}` - Get partner by company name (ADR-003)
   - `POST /api/v1/partners` - Create new partnership (organizer-only)
   - `PATCH /api/v1/partners/{companyName}` - Update partnership details
   - `DELETE /api/v1/partners/{companyName}` - Deactivate partnership (soft delete)

3. **Partner Contact Management** (ADR-003/004 pattern)
   - `GET /api/v1/partners/{companyName}/contacts` - List contacts with User data from HTTP API
   - `POST /api/v1/partners/{companyName}/contacts` - Add contact (stores username)
   - `DELETE /api/v1/partners/{companyName}/contacts/{username}` - Remove contact
   - Contact roles: PRIMARY, BILLING, TECHNICAL, MARKETING
   - Business rule: At least one PRIMARY contact required
   - **PartnerContact stores `username` (String), NOT `userId` (UUID)**
   - **User data fetched via HTTP call to User Service API**

4. **Topic Voting**
   - `GET /api/v1/partners/{companyName}/votes` - List partner's topic votes
   - `POST /api/v1/partners/{companyName}/votes` - Cast vote on topic (1-5 scale)
   - Vote weight based on partnership tier (BRONZE=1, STRATEGIC=5)
   - One vote per topic per partner (constraint)

5. **Topic Suggestions**
   - `GET /api/v1/partners/{companyName}/suggestions` - List suggestions
   - `POST /api/v1/partners/{companyName}/suggestions` - Submit new topic suggestion
   - Suggestion status: SUBMITTED, UNDER_REVIEW, ACCEPTED, REJECTED, IMPLEMENTED

6. **OpenAPI documentation**
   - OpenAPI 3.1 spec at `docs/api/partners-api.openapi.yml`
   - ErrorResponse and PaginationMetadata imported from shared-kernel (NOT defined locally)
   - All endpoints documented with request/response schemas

7. **Database schema with Flyway migrations**
   - PostgreSQL schema uses meaningful IDs (companyName, username)
   - Flyway migration in `shared-kernel/src/main/resources/db/migration/`
   - All constraints, indexes (NO foreign keys to Company/User tables)

8. **Domain events publishing**
   - PartnerCreatedEvent when partnership created
   - PartnerUpdatedEvent when partnership modified
   - PartnerContactAddedEvent when contact added
   - TopicVoteSubmittedEvent when vote cast
   - All events include full metadata (correlationId, causationId, userId)

9. **Shared-Kernel integration** (mandatory)
   - Exceptions extend shared-kernel hierarchy (PartnerNotFoundException extends NotFoundException)
   - GlobalExceptionHandler uses `ch.batbern.shared.dto.ErrorResponse`
   - Domain events extend `DomainEvent<UUID>`
   - Inject DomainEventPublisher from shared-kernel

10. **OpenAPI Code Generation - Hybrid Contract-First** (mandatory)
    - OpenAPI spec is source of truth: `docs/api/partners-api.openapi.yml`
    - Generated API interfaces in `build/generated/src/main/java/ch/batbern/partners/api/generated/`
      - `PartnersApi.java` - Partner CRUD operations
      - `PartnerContactsApi.java` - Contact management
      - `TopicVotingApi.java` - Topic voting and suggestions
    - Generated DTOs in `build/generated/src/main/java/ch/batbern/partners/dto/generated/`
      - `PartnerResponse.java`, `UpdatePartnerRequest.java`
      - `PartnerContactResponse.java`, `AddPartnerContactRequest.java`
      - `TopicVote.java`, `CastVoteRequest.java`
      - `TopicSuggestion.java`, `SubmitSuggestionRequest.java`
    - Shared-kernel types imported (NOT generated):
      - `ch.batbern.shared.dto.ErrorResponse`
      - `ch.batbern.shared.api.PaginationMetadata`
      - `ch.batbern.shared.dto.PaginatedResponse<T>`
    - Controllers implement generated interfaces manually
    - Build fails if implementation doesn't match spec
    - Code regenerated on every build (in `build/generated/`, gitignored)

11. **HTTP Client Integration** (ADR-003 microservices pattern)
    - CompanyServiceClient for `GET /api/v1/companies/{companyName}`
    - UserServiceClient for `GET /api/v1/users/{username}`
    - Circuit breaker with Resilience4j (fallback to cached data)
    - HTTP client responses cached (5min TTL)
    - Handle 404 errors gracefully (company/user not found)

12. **Integration tests**
    - All tests extend AbstractIntegrationTest (PostgreSQL via Testcontainers)
    - Test partner CRUD operations
    - Test contact management with mocked User Service HTTP responses
    - Test topic voting with weight calculations
    - Test business rules (minimum primary contact, vote uniqueness)
    - Mock HTTP clients for Company/User Service calls in tests
    - Coverage >80%

13. **Performance**
    - Partner list: <100ms P95
    - Partner detail: <150ms P95
    - Partner with contacts (HTTP enrichment): <300ms P95 (allows for HTTP call overhead)

### Quick Reference: Key Patterns

**Partner Entity (Meaningful ID):**
```java
@Entity
@Table(name = "partners")
public class Partner {
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private UUID id;  // ✅ UUID PK for this service

    @Column(name = "company_name", nullable = false, unique = true, length = 12)
    private String companyName;  // ✅ ADR-003: meaningful ID, NOT UUID FK

    // NO: private UUID companyId;  ❌ WRONG
}
```

**HTTP Enrichment Pattern:**
```java
@Service
@RequiredArgsConstructor
public class PartnerService {
    private final PartnerRepository partnerRepository;
    private final CompanyServiceClient companyServiceClient;  // HTTP client

    public PartnerResponse getPartner(String companyName, Set<String> includes) {
        Partner partner = partnerRepository.findByCompanyName(companyName)
            .orElseThrow(() -> new NotFoundException("Partner not found"));

        PartnerResponse response = mapToResponse(partner);

        // HTTP enrichment per ADR-004
        if (includes.contains("company")) {
            CompanyResponse company = companyServiceClient.getCompany(companyName);
            response.setCompany(company);  // Enrich response
        }

        return response;
    }
}
```

**Controller Pattern:**
```java
@RestController
@RequiredArgsConstructor
public class PartnerController implements PartnersApi {  // Generated interface
    private final PartnerService partnerService;
    private final IncludeParser includeParser;

    @Override
    public ResponseEntity<PartnerResponse> getPartnerByCompanyName(
            String companyName,
            Optional<String> include) {

        Set<String> includes = includeParser.parse(include.orElse(null));
        PartnerResponse partner = partnerService.getPartner(companyName, includes);

        return ResponseEntity.ok(partner);
    }
}
```
