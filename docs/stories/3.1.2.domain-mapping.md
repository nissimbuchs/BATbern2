## Architecture Context

### Architecture Patterns
- **Data Migration Pattern**: Extract-Transform-Load (ETL) with domain-driven schema mapping
- **Reference Pattern (ADR-004)**: Entity references use foreign keys to related entities without field duplication
  - Speaker в†’ User (via userId FK): bio, photo, company stored in User (NOT duplicated in Speaker)
  - SessionUser в†’ User (via userId FK): username, email, firstName, lastName from User
  - PartnerContact в†’ User (via userId FK): contact person fields from User
- **Meaningful IDs (Story 1.16.2)**:
  - Database: UUID primary keys for performance
  - API: Meaningful string identifiers (eventCode, company name, username)
  - URLs: `/events/BATbern56`, `/companies/GoogleZH`, `/speakers/john.doe`
- **File Storage (ADR-002)**: Generic File Upload Service for logos/photos with S3 and CloudFront CDN
- **Database Schema Source of Truth**: `docs/architecture/03-data-architecture.md`

**Company User Management Service** (`services/company-user-management-service/src/main/resources/db/migration/`):
- `V3__Create_companies_schema.sql` - Companies table:
  - `name VARCHAR(255) NOT NULL UNIQUE` (meaningful ID, public identifier)
  - `display_name VARCHAR(255)` (full official name)
  - `website VARCHAR(500)`
  - `logo_url VARCHAR(1000)` (CloudFront CDN URL)
  - `logo_s3_key VARCHAR(500)` (S3 storage key)
  - `logo_file_id VARCHAR(100)` (file identifier)
  - `is_verified BOOLEAN NOT NULL DEFAULT FALSE`

- `V4__Create_user_profiles_table.sql` - User profiles table:
  - `id UUID PRIMARY KEY` (internal database key, NOT exposed in API)
  - `username VARCHAR(100) NOT NULL UNIQUE` (public API identifier per Story 1.16.2)
  - `company_id VARCHAR(12)` (FK to `companies.name`, NOT UUID per Story 1.16.2)
  - `bio TEXT` (single source of truth per ADR-004)
  - `profile_picture_url VARCHAR(2048)` (CloudFront CDN URL)
  - `profile_picture_s3_key VARCHAR(500)` (S3 storage key)
  - Constraints: `chk_company_id_format CHECK (company_id ~ '^[a-zA-Z0-9]{1,12}$')`

**вљ пёЏ SCHEMA INCONSISTENCY DETECTED**:
- `companies.name` is `VARCHAR(255)` (allows up to 255 chars)
- `user_profiles.company_id` has constraint `VARCHAR(12)` with regex check limiting to 12 chars
- **Migration Strategy**: Map company identifiers to max 12 chars to satisfy the stricter constraint
  - Examples: `"sbb"` (3 chars), `"mobiliar"` (8 chars), `"swisscom"` (8 chars) вњ…
  - If legacy company name > 12 chars, truncate or abbreviate during migration
  - This aligns with Story 1.16.2 meaningful ID pattern (short, memorable identifiers)

**Event Management Service** (`services/event-management-service/src/main/resources/db/migration/`):
- `V2__Create_events_schema.sql` - Events and sessions tables:
  - `events` table:
    - `id UUID PRIMARY KEY` (internal database key)
    - `event_number INTEGER UNIQUE NOT NULL` (sequential number)
    - `title VARCHAR(255) NOT NULL`
    - `event_date TIMESTAMP WITH TIME ZONE NOT NULL`
    - `status VARCHAR(50)` (CHECK constraint with values)
  - `sessions` table:
    - `id UUID PRIMARY KEY`
    - `event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE`
    - `title VARCHAR(255) NOT NULL`
    - `description TEXT`
    - `session_type VARCHAR(50)` (CHECK constraint)

- `V7__Add_session_users_junction_table.sql` - Session-User junction:
  - `session_users` table (many-to-many, ADR-004 pattern):
    - `session_id UUID NOT NULL REFERENCES sessions(id) ON DELETE CASCADE`
    - `user_id UUID NOT NULL` (FK to user_profiles.id in company-user-management-service)
    - `speaker_role VARCHAR(50)` (primary_speaker, co_speaker, moderator, panelist)
    - Constraint: `UNIQUE (session_id, user_id)`

**Schema Validation During Migration**:
- All mapper tests MUST validate against actual Flyway migration column names, types, and constraints
- Use `VARCHAR(12)` for company_id (not unlimited string)
- Use `VARCHAR(100)` for username (not unlimited string)
- Respect CHECK constraints (event status, session type, speaker role)
- Follow ADR-004: User fields in user_profiles, not duplicated in domain entities

## Acceptance Criteria

1. **Company Mapping**: Map company data to Company Management Service schema (FIRST - prerequisite)
   - Normalize 143 company variations to 70 unique companies (per migration-readiness-assessment.md)
   - Use `docs/migration/companies.json` as authoritative source (metadata: 42 complete, 53 with URLs, 25 with local logos)
   - Map `companies.json` в†’ `id` field to `Company.name` (meaningful ID per Story 1.16.2)
   - Map `companies.json` в†’ `displayName` to `Company.displayName` for full official name
   - Map `companies.json` в†’ `url` to `Company.website`
   - Map `companies.json` в†’ `logo`/`logoFilePath` to S3 key structure for `Company.logoS3Key`
   - Handle companies with `logoUrl` (online) vs `logo` (local file) appropriately
   - Process companies by batch priority (Top 5, Priority, Standard, Low Priority)
   - **Output**: 70 Company entities created, ready for User.companyId foreign key references

2. **Event Mapping**: Map legacy event JSON (`topics.json`) to Event Management Service schema
   - Map `bat` number to `eventNumber` and auto-generate `eventCode` (e.g., `BATbern56`)
   - Map `topic` to Event.title
   - Parse `datum` to Event.eventDate (handle multiple German date formats)
   - Map `eventType` ("Abend-BAT", "Ganztag-BAT") to Event.eventType enum
   - Define Event.status and Event.workflowState for historical events (all ARCHIVED/COMPLETED)

3. **Speaker Mapping**: Map speaker profiles to Speaker Coordination Service schema (AFTER companies)
   - Extract unique speakers from `sessions.json` в†’ `referenten[]` array
   - Normalize speaker name format: "FirstName LastName, Company" в†’ separate fields
   - Map `bio` to User.bio (single source of truth per ADR-004)
   - Map `portrait` filename to S3 key structure for User.profilePictureUrl
   - Map `company` identifier to Company.name (normalized from 143 variations to 70 unique)
   - **Validate**: Each User.companyId references an existing Company.name from AC1
   - Create Speaker entity with userId FK (ADR-004 pattern)
   - **Dependency**: Requires Company entities from AC1 to exist before creating Users

4. **File Mapping**: Map file paths to S3 bucket structure and CDN URLs
   - **Company Logos**: `companies.json` в†’ `logo`/`logoUrl` в†’ S3 key: `logos/{year}/companies/{companyName}/logo-{fileId}.{ext}`
   - **Profile Pictures**: `referenten[].portrait` в†’ S3 key: `profile-pictures/{year}/{username}/profile-{fileId}.{ext}`
   - **Presentations**: `sessions.json` в†’ `pdf` field в†’ S3 key: `presentations/{eventNumber}/{filename}` (custom pattern)
   - **Event Photos**: `pictures.json` в†’ `image` в†’ S3 key: `photos/events/{eventNumber}/{filename}` (custom pattern)
   - CloudFront URL pattern: `https://cdn.batbern.ch/{s3-key}` (production/staging)
   - MinIO URL pattern: `http://localhost:8450/{bucketName}/{s3-key}` (local development)

5. **Relationship Mapping**: Define foreign key relationships across services
   - Session в†’ Event: `session.eventId` = Event.id (UUID, within same service)
   - SessionUser в†’ User: `session_user.userId` = User.id (UUID FK, cross-service)
   - SessionUser в†’ Session: `session_user.sessionId` = Session.id (UUID, within same service)
   - Speaker в†’ User: `speaker.userId` = User.id (UUID FK, cross-service, ADR-004)
   - User в†’ Company: `user.companyId` = Company.name (meaningful ID, not UUID per Story 1.16.2)
   - SpeakingEngagement в†’ Speaker: `engagement.speakerId` = Speaker.id (UUID, within same service)

### Mapping Rules

#### Event Mapping Rules

1. **BAT Number в†’ Event Code**
   - Input: `topics.json` в†’ `bat: 56`
   - Output: `Event.eventCode = "BATbern56"`, `Event.eventNumber = 56`
   - Rule: `eventCode = "BATbern" + bat.toString()`

2. **Date Parsing** (Handle Multiple Formats)
   - Format 1: `"24. Juni 05, 16:00h - 18:30h"` в†’ `2005-06-24T16:00:00Z`
   - Format 2: `"Donnerstag, 2. Mai 2024, 16:00 - 18:30 Uhr"` в†’ `2024-05-02T16:00:00Z`
   - Format 3: `"2024-05-02"` в†’ `2024-05-02T00:00:00Z`
   - Use date-fns or moment for parsing with locale support (German months)

3. **Event Type Mapping**
   - `"Abend-BAT"` в†’ `EventType.EVENING`
   - `"Ganztag-BAT"` в†’ `EventType.FULL_DAY`
   - `"Halb-BAT"` в†’ `EventType.AFTERNOON`
   - Default: `EventType.EVENING`

4. **Historical Event Status**
   - All historical events: `Event.status = EventStatus.ARCHIVED`
   - All historical events: `Event.workflowState = EventWorkflowState.PUBLISHED`

#### Speaker Mapping Rules

1. **Name Parsing**
   - Input: `"Thomas Goetz, Die Mobiliar"`
   - Parse pattern: `/^(.+?),\s*(.+)$/`
   - Output: `firstName = "Thomas"`, `lastName = "Goetz"`, `company = "mobiliar"`
   - Normalize company to match `companies.json` identifiers

2. **Username Generation**
   - Pattern: `firstName.toLowerCase() + "." + lastName.toLowerCase()`
   - Example: `"Thomas Goetz"` в†’ `"thomas.goetz"`
   - Handle duplicates: append number (`"thomas.goetz2"`)

3. **Bio Mapping (ADR-004)**
   - Source: `sessions.json` в†’ `referenten[].bio`
   - Target: `User.bio` (NOT in Speaker entity)
   - Speaker entity only contains domain-specific fields

4. **Portrait Mapping (Following ProfilePictureService pattern)**
   - Source: `sessions.json` в†’ `referenten[].portrait` = `"thomas.goetz.jpg"`
   - Pattern: `profile-pictures/{year}/{username}/profile-{fileId}.{ext}` (from ProfilePictureService)
   - Year: Migration year (2025)
   - FileId: Generate UUID for each portrait during migration
   - Example: `thomas.goetz.jpg` в†’ `profile-pictures/2025/thomas.goetz/profile-{uuid}.jpg`
   - CloudFront URL: `https://cdn.batbern.ch/profile-pictures/2025/thomas.goetz/profile-{uuid}.jpg`
   - Target: `User.profilePictureUrl` and `User.profilePictureS3Key`
   - Reference: ProfilePictureService.java line 161

#### Company Mapping Rules

1. **Company Data Source**
   - Source: `docs/migration/companies.json` (70 companies, 21KB)
   - Metadata: 42 complete (60%), 53 with URLs (76%), 25 with local logos
   - Process by batch priority (Batches 1-4 defined in metadata):
     - **Batch 1**: Top 5 companies (10+ speakers) - 100% complete (SBB, Mobiliar, Swisscom, PostFinance, ELCA)
     - **Batch 2**: Priority companies (5-9 speakers) - 100% complete (RTC, BKW, ipt, mtrail)
     - **Batch 3**: Standard priority (2-4 speakers) - 55% complete (17/31)
     - **Batch 4**: Low priority (1 speaker) - 55% complete (16/29)
   - Skip companies with `status: "duplicate"` (merge with original company)
   - Refer to `docs/migration/COLLECTION-REPORT.md` for detailed company data collection methodology

2. **Company Identifier Mapping**
   - Source: `companies.json` в†’ `id` field
   - Target: `Company.name` (meaningful ID per Story 1.16.2)
   - Examples: `"sbb"`, `"mobiliar"`, `"swisscom"`, `"postfinance"`, `"elca"`
   - API URLs: `/companies/sbb` (NOT `/companies/{uuid}`)

3. **Display Name and URL Mapping**
   - Source: `companies.json` в†’ `displayName` field
   - Target: `Company.displayName`
   - Examples: `"SBB CFF FFS"`, `"Die Mobiliar"`, `"Swisscom AG"`
   - Source: `companies.json` в†’ `url` field
   - Target: `Company.website`

4. **Logo Mapping Strategies**

   **Strategy A: Local Logo Files** (25 companies with `logo` + `logoFilePath`)
   - Source: `companies.json` в†’ `logo` = `"sbb.jpg"`, `logoFilePath` = absolute path
   - Pattern: `logos/{year}/companies/{companyName}/logo-{fileId}.{ext}` (from GenericLogoService)
   - Example: `sbb.jpg` в†’ S3 key `logos/2025/companies/sbb/logo-{uuid}.jpg`
   - CloudFront URL: `https://cdn.batbern.ch/logos/2025/companies/sbb/logo-{uuid}.jpg`
   - Note: Follow existing CompanyService pattern (line 369: `logos/%d/companies/%s/logo-%s.%s`)

   **Strategy B: Online Logo URLs** (8 companies with `logoUrl` only)
   - Source: `companies.json` в†’ `logoUrl` = online SVG/PNG URL
   - Download logo from URL during migration
   - Generate unique fileId (UUID) for each logo
   - Upload to S3 Key: `logos/{year}/companies/{companyName}/logo-{fileId}.{ext}`
   - Example: Puzzle logoUrl в†’ download в†’ `logos/2025/companies/puzzle/logo-{uuid}.svg`

   **Strategy C: Missing Logos** (companies with `status: "needs_logo"`)
   - Create placeholder or skip logo upload
   - Set `Company.logoUrl = null`, `Company.logoS3Key = null`
   - Can be backfilled post-migration

5. **Duplicate Handling**
   - Companies with `status: "duplicate"` should NOT create new entities
   - Map references to the original company ID
   - Examples: `logo-pf` в†’ `postfinance`, `aws-logo` в†’ `aws`, `ibm-8bar-logo` в†’ `ibm`

6. **Status-Based Processing**
   - `"complete"`: Has both logo and URL, migrate immediately
   - `"needs_logo"`: Has URL only, use placeholder logo
   - `"needs_url"`: Has logo only, URL can be backfilled
   - `"pending_url"`: Logo exists locally, URL pending collection
   - `"duplicate"`: Skip, map to original company
   - `"needs_clarification"`: Manual review required

   - Foreign Key: `Speaker.userId` = `User.id` (UUID)

   - Foreign Key: `User.companyId` = `Company.name` (string, NOT UUID)
   - Example: `User.companyId = "mobiliar"` references `Company.name = "mobiliar"`
