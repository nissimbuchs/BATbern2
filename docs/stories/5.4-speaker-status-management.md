# Story 5.4: Speaker Status Management

## Status
Done

## Story

**As an** organizer,
**I want** to track speaker status transitions,
**so that** I know which speakers have accepted, declined, or are pending response.

## Domain Context

### Primary Domain

**Event Management Domain** - Organizer workflows and speaker status tracking

This story implements **Workflow Step 5** of the 16-step event workflow (FR2):
- Speaker Status Tracking (OPEN → CONTACTED → READY → ACCEPTED/DECLINED)
- Manual status updates by organizers
- Status change history and audit trail
- Visual status dashboard with progress tracking

### Involved Services

**Services:**
- **Speaker Coordination Service**: Status management API and business logic (controllers, services)
- **Event Management Service**: Data persistence layer (speaker_status_history table, repositories)
- **API Gateway**: Routes speaker status endpoints

**Architecture Note**:
- **Data Layer**: `speaker_status_history` table resides in **event-management-service** database (colocated with speaker_pool from Story 5.2)
- **Service Layer**: Speaker-coordination-service provides the API and business logic
- **Workflow State**: Stored in `speaker_pool.status` column (existing from Story 5.2), history tracked separately
- **Rationale**: Colocating related data simplifies FK constraints and transactions, while service boundaries maintain domain separation

### Status Workflow

1. **Status Transitions**: Follow workflow sequence: OPEN → CONTACTED → READY → ACCEPTED/DECLINED (can skip READY if declining directly)
2. **Manual Status Updates**: Organizer manually updates status via dropdown or drag-and-drop to status lane
3. **Status Change Timestamp**: Track when each status change occurred with `changed_at` timestamp
4. **Status Change Reason**: Optional notes field for status changes (max 2000 characters)
5. **Status Dashboard**: Visual dashboard showing speakers grouped by status with counts per lane
6. **Acceptance Rate Tracking**: Show percentage of speakers who accepted vs declined vs pending

### Status Indicators

7. **Color Coding**: Visual color coding for each status:
   - Green: ACCEPTED
   - Red: DECLINED
   - Yellow: CONTACTED, READY
   - Gray: OPEN
8. **Progress Bar**: Overall event progress based on speaker statuses (e.g., "6/8 speakers accepted")
9. **Notifications**: Notify organizers when speaker status changes (if multiple organizers assigned to event)

### Workflow Engine Integration

10. **State Transition**: Manual status updates call `speakerWorkflowService.updateSpeakerWorkflowState(sessionId, speakerId, newState, organizerId)`
11. **Event Listener**: Listen for `SpeakerWorkflowStateChangeEvent` to update UI in real-time via React Query invalidation
12. **Validation**: Enforce valid state transitions:
    - OPEN → CONTACTED ✓
    - CONTACTED → READY ✓
    - CONTACTED → DECLINED ✓
    - READY → ACCEPTED ✓
    - READY → DECLINED ✓
    - ACCEPTED → DECLINED ✗ (cannot un-accept)
    - DECLINED → ACCEPTED ✗ (cannot un-decline)
13. **Acceptance Check**: When speaker moves to ACCEPTED, trigger overflow detection via `EventWorkflowStateMachine.checkForOverflow(eventId)`

### Status Transition Validation

**Valid Transitions**:
```java
// In SpeakerStatusService

private static final Map<SpeakerWorkflowState, Set<SpeakerWorkflowState>> VALID_TRANSITIONS = Map.of(
    OPEN, Set.of(CONTACTED),
    CONTACTED, Set.of(READY, DECLINED),
    READY, Set.of(ACCEPTED, DECLINED),
    ACCEPTED, Set.of(SLOT_ASSIGNED),  // Next story
    DECLINED, Set.of()  // Terminal state
);

public void validateTransition(SpeakerWorkflowState from, SpeakerWorkflowState to) {
    if (!VALID_TRANSITIONS.getOrDefault(from, Set.of()).contains(to)) {
        throw new InvalidStateTransitionException(
            String.format("Invalid transition from %s to %s", from, to)
        );
    }
}
```

### Workflow Integration

```java
// In SpeakerStatusService.updateStatus()

@Transactional
public SpeakerStatusResponse updateStatus(
    String eventCode,
    UUID speakerId,
    String organizerUsername,
    UpdateStatusRequest request
) {
    // 1. Validate speaker exists
    SpeakerPool speaker = speakerPoolRepository.findById(speakerId)
        .orElseThrow(() -> new NotFoundException("Speaker not found: " + speakerId));

    SpeakerWorkflowState currentStatus = speaker.getStatus();
    SpeakerWorkflowState newStatus = request.getNewStatus();

    // 2. Validate state transition
    validateTransition(currentStatus, newStatus);

    // 3. Create status history record
    SpeakerStatusHistory history = new SpeakerStatusHistory();
    history.setSpeakerPoolId(speakerId);
    history.setSessionId(speaker.getSessionId());
    history.setEventCode(eventCode);
    history.setPreviousStatus(currentStatus);
    history.setNewStatus(newStatus);
    history.setChangedByUsername(organizerUsername);
    history.setChangeReason(request.getReason());
    history.setChangedAt(Instant.now());

    speakerStatusHistoryRepository.save(history);

    // 4. Transition speaker workflow state (delegates to Story 5.1a workflow service)
    speakerWorkflowService.updateSpeakerWorkflowState(
        speaker.getSessionId(),
        speakerId,
        newStatus,
        organizerUsername
    );

    // 5. Check for overflow if accepting speaker
    if (newStatus == SpeakerWorkflowState.ACCEPTED) {
        checkForOverflow(eventCode);
    }

    // 6. Publish domain event (DomainEventPublisher from shared-kernel)
    // Note: SpeakerWorkflowService already publishes SpeakerWorkflowStateChangeEvent
    // We can publish additional business event if needed
    domainEventPublisher.publishEvent(new SpeakerStatusChangedEvent(
        eventCode,
        speakerId,
        currentStatus,
        newStatus,
        organizerUsername,
        history.getChangedAt()
    ));

    return mapToResponse(speaker, history);
}

private void checkForOverflow(String eventCode) {
    Event event = eventRepository.findByEventCode(eventCode)
        .orElseThrow(() -> new NotFoundException("Event not found"));

    int maxSlots = event.getSlotConfiguration().getMaxSlots();
    long acceptedCount = speakerPoolRepository.countByEventCodeAndStatus(
        eventCode,
        SpeakerWorkflowState.ACCEPTED
    );

    if (acceptedCount > maxSlots) {
        domainEventPublisher.publishEvent(new SpeakerOverflowDetectedEvent(
            eventCode,
            acceptedCount,
            maxSlots
        ));
    }
}
```

### Status Dashboard Summary Calculation

```java
// In SpeakerStatusService.getStatusSummary()

public StatusSummaryResponse getStatusSummary(String eventCode) {
    List<SpeakerPool> speakers = speakerPoolRepository.findByEventCode(eventCode);

    Map<SpeakerWorkflowState, Long> counts = speakers.stream()
        .collect(Collectors.groupingBy(
            SpeakerPool::getStatus,
            Collectors.counting()
        ));

    long totalSpeakers = speakers.size();
    long acceptedCount = counts.getOrDefault(ACCEPTED, 0L);
    long declinedCount = counts.getOrDefault(DECLINED, 0L);
    long pendingCount = totalSpeakers - acceptedCount - declinedCount;

    double acceptanceRate = totalSpeakers > 0
        ? (double) acceptedCount / totalSpeakers * 100
        : 0.0;

    Event event = eventRepository.findByEventCode(eventCode)
        .orElseThrow(() -> new NotFoundException("Event not found"));

    int minSlots = event.getSlotConfiguration().getMinSlots();
    int maxSlots = event.getSlotConfiguration().getMaxSlots();

    return StatusSummaryResponse.builder()
        .eventCode(eventCode)
        .statusCounts(counts)
        .totalSpeakers(totalSpeakers)
        .acceptedCount(acceptedCount)
        .declinedCount(declinedCount)
        .pendingCount(pendingCount)
        .acceptanceRate(acceptanceRate)
        .minSlotsRequired(minSlots)
        .maxSlotsAllowed(maxSlots)
        .thresholdMet(acceptedCount >= minSlots)
        .overflowDetected(acceptedCount > maxSlots)
        .build();
}
```

### Drag-and-Drop Implementation with @dnd-kit

```typescript
// web-frontend/src/components/organizer/SpeakerStatus/SpeakerStatusLanes.tsx

import { DndContext, DragEndEvent, closestCenter } from '@dnd-kit/core';
import { SortableContext, verticalListSortingStrategy } from '@dnd-kit/sortable';

const SpeakerStatusLanes: React.FC<Props> = ({ eventCode, speakers, onStatusChange }) => {
  const statusLanes = ['OPEN', 'CONTACTED', 'READY', 'ACCEPTED', 'DECLINED'];

  const handleDragEnd = async (event: DragEndEvent) => {
    const { active, over } = event;

    if (over && active.id !== over.id) {
      const speakerId = active.id as string;
      const newStatus = over.id as SpeakerWorkflowState;

      // Open confirmation dialog
      setStatusChangeDialog({
        speaker: speakers.find(s => s.id === speakerId),
        newStatus,
        onConfirm: async (reason: string) => {
          await onStatusChange(speakerId, newStatus, reason);
        },
      });
    }
  };

  return (
    <DndContext collisionDetection={closestCenter} onDragEnd={handleDragEnd}>
      <Grid container spacing={2}>
        {statusLanes.map(status => (
          <Grid item xs={12} md={2.4} key={status}>
            <StatusLane
              status={status}
              speakers={speakers.filter(s => s.status === status)}
            />
          </Grid>
        ))}
      </Grid>
    </DndContext>
  );
};
```
