# Story 6.4: Speaker Dashboard (View-Only)

Status: ready-for-dev

## Story

As a **speaker**,
I want to view my upcoming and past BATbern presentations through a self-service dashboard,
so that I can see my speaking history, event details, submission status, and deadlines without contacting organizers.

## Domain Context

### Primary Domain

Speaker Coordination Domain - Dashboard & speaker engagement visibility

### Involved Services

- **Event Management Service**: Dashboard summary endpoint aggregating speaker's events, sessions, content status, materials
- **API Gateway**: Route `/api/v1/speaker-portal/dashboard`
- **Company User Management Service**: User profile data enrichment (via HTTP client, ADR-004)

### Cross-Domain Dependencies

- Story 6.1a (Magic Link Infrastructure) - Token validation and session management
- Story 6.2b (Profile Update) - Profile data, photo display
- Story 6.3 (Content Submission) - Content status and material info
- Story 5.4 (Speaker Status Dashboard) - Organizer-side counterpart (DO NOT reuse for speaker-facing view)

## Acceptance Criteria

### AC1: Dashboard Access via Magic Link

1. Speaker accesses dashboard via URL: `/speaker-portal/dashboard?token={token}`
2. Token validated using existing `MagicLinkService` (from Story 6.1a)
3. Invalid/expired token shows clear error with instructions to request new link
4. Dashboard route added to `App.tsx` as public route (same pattern as existing speaker-portal routes)

### AC2: Upcoming Events View

1. Dashboard shows upcoming events where speaker has `workflow_state` in (INVITED, ACCEPTED, CONTENT_SUBMITTED, QUALITY_REVIEWED, SLOT_ASSIGNED, CONFIRMED)
2. Each event card displays:
   - Event name and date
   - Session title (if assigned)
   - Speaker's workflow state with friendly labels:
     - "Invitation Pending" (INVITED)
     - "Accepted" (ACCEPTED)
     - "Content Submitted" (CONTENT_SUBMITTED) — *added 2026-02-11*
     - "Quality Reviewed" (QUALITY_REVIEWED) — *added 2026-02-11*
     - "Slot Assigned" (SLOT_ASSIGNED) — *added 2026-02-11*
     - "Confirmed" (CONFIRMED)
   - Content status (friendly label: "Not Submitted", "Under Review", "Approved", "Revision Needed")
   - Material upload status (uploaded/not uploaded)
   - Response deadline (if INVITED, with visual urgency indicator)
   - Content deadline (if ACCEPTED, with visual urgency indicator)
3. Events sorted by event date ascending (soonest first)
4. Quick-action links per event: "Respond", "Update Profile", "Submit Content" (contextual based on state)

### AC3: Past Events View

1. Dashboard shows past events where event date < today AND speaker was CONFIRMED or ACCEPTED
2. Each past event card displays:
   - Event name and date
   - Session title and topic
   - Whether presentation materials are available
3. Past events sorted by date descending (most recent first)
4. Empty state: "No past speaking engagements yet"

### AC4: Material Status Overview

1. For each upcoming event, show material submission status:
   - Title: submitted or pending
   - Abstract: submitted or pending (with char count)
   - Presentation file: uploaded or not
2. If `content_status` = `REVISION_NEEDED`, show reviewer feedback prominently
3. Link to content submission page for incomplete items

### AC5: Organizer Contact Information

1. Display organizer contact for each event
2. Show organizer name and email
3. "Contact Organizer" link opens email with pre-filled subject line

### AC6: Magic Link Session Management

1. Dashboard magic link tokens use `VIEW` action type (reusable, not single-use)
2. Token expiry: 30 days (configurable, same as existing tokens)
3. Expired token: show message "Your link has expired. Please contact the event organizer for a new link."
4. No account creation required - magic link is the only auth mechanism

### AC7: Mobile Responsive Design

1. Dashboard adapts to mobile viewport (375px minimum)
2. Event cards stack vertically on mobile
3. Touch-friendly action buttons (minimum 44px tap target)
4. Readable typography at all breakpoints

### AC8: Empty & Loading States

1. Loading: skeleton placeholders while data loads
2. No upcoming events: "You don't have any upcoming speaking engagements"
3. API error: friendly error message with retry button
4. Token loading state during validation

## Dev Notes

### Architecture Patterns & Constraints

**Service**: Event Management Service (`services/event-management-service/`)
- Dashboard endpoint lives here because SpeakerPool, Events, Sessions are all in this service's database
- Organizer contact info enriched via `UserApiClient` HTTP call (ADR-004 cross-service pattern)
- Use `MagicLinkService.validateToken()` for token auth (from Story 6.1a)

**Authentication**: Magic link token, NOT JWT/Cognito
- All speaker portal endpoints use `Skip-Auth: true` header pattern
- Token passed as query parameter: `?token={token}`
- Validate with `MagicLinkService` - returns `SpeakerPool` association

**Security Config**: Must add new endpoint to BOTH:
- `api-gateway/.../SecurityConfig.java` - permit `/api/v1/speaker-portal/dashboard`
- `event-management-service/.../SecurityConfig.java` - permit same path
- **CRITICAL**: Previous stories (6.2b, 6.3) had bugs from missing this in one of the two configs. Add to BOTH.

### Database Queries (Backend)

**Upcoming events**: Speaker pool entries where event date >= today
```sql
SELECT sp.*, e.*, s.*
FROM speaker_pool sp
JOIN events e ON sp.event_id = e.id
LEFT JOIN sessions s ON sp.session_id = s.id
WHERE sp.username = :username
  AND e.event_date >= CURRENT_DATE
  AND sp.status IN ('INVITED', 'ACCEPTED', 'CONTENT_SUBMITTED', 'QUALITY_REVIEWED', 'SLOT_ASSIGNED', 'CONFIRMED')
ORDER BY e.event_date ASC
```

**Past events**: Speaker pool entries where event date < today
```sql
SELECT sp.*, e.*, s.*
FROM speaker_pool sp
JOIN events e ON sp.event_id = e.id
LEFT JOIN sessions s ON sp.session_id = s.id
WHERE sp.username = :username
  AND e.event_date < CURRENT_DATE
  AND sp.status IN ('ACCEPTED', 'CONFIRMED')
ORDER BY e.event_date DESC
```

**Organizer enrichment**: For each event, get organizer username from `events.created_by` or designated organizer field, then call `UserApiClient.getUser(username)` for name/email.
