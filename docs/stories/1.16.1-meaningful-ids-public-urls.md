# Story 1.16.1: Meaningful IDs for Public-Facing URLs

## Status
**Draft** - Awaiting approval

## Story

**As a** BATbern platform user (attendee, organizer, or partner),
**I want** meaningful, human-readable URLs for events, companies, and user profiles,
**so that** I can easily share links, remember URLs, and navigate the platform intuitively without dealing with cryptic UUIDs.

## Domain Context

### Primary Domain
**Shared/Infrastructure** - Cross-cutting concern affecting all bounded contexts:
- Event Management Domain (event codes for public sharing)
- Company Management Domain (company name identifiers)
- User Management Domain (username-based profiles)
- Speaker Coordination Domain (uses user identities)
- Attendee Experience Domain (content discovery via friendly URLs)

### Involved Services
- **All Microservices:** This is a shared-kernel infrastructure change
  - Event Management Service (event codes)
  - Company User Management Service (company names, usernames)
  - Speaker Coordination Service (uses usernames for speakers)
  - Partner Coordination Service (uses company names and usernames)
  - Attendee Experience Service (session slugs, event codes)
  - API Gateway (routing updates for friendly IDs)

### Cross-Domain Dependencies
- **Shared Kernel:** ID value objects (EventId, CompanyId, UserId, SessionId)
- **Database Layer:** All service databases need schema migrations
- **API Gateway:** URL routing patterns updated for friendly IDs
- **Frontend:** React Router and API service layer updates

## Requirements Context

### Related Functional Requirements
- **FR-Infrastructure-001:** Platform must support user-friendly, shareable URLs
- **FR-SEO-001:** Public-facing pages must have SEO-friendly URL structures
- **FR-UX-001:** URLs should be intuitive and memorable for end users
- **FR-API-001:** API endpoints must support both UUID and friendly ID lookups (transition period)

### Workflow Steps (if applicable)
Not directly tied to the 16-step event workflow, but affects:
- **All Steps:** Event management screens will use `BATbern{number}` in URLs
- **Steps 2-4:** Speaker coordination uses `firstname.lastname` identifiers

## Acceptance Criteria

### Events (AC 1-3)

1. **AC1:** When creating a new event, the system automatically generates an event code in format `BATbern{eventNumber}` where `eventNumber` is the sequential event number
   - Event code is unique across all events
   - Event code is stored in `event_code` column
   - Event code is returned in API responses as the primary identifier

2. **AC2:** API endpoints accept event code as path parameter
   - `GET /api/v1/events/{eventCode}` returns event by code (e.g., `BATbern56`)
   - `PUT /api/v1/events/{eventCode}` updates event by code
   - `DELETE /api/v1/events/{eventCode}` deletes event by code
   - Backend resolves event code to UUID internally

3. **AC3:** Frontend uses event codes in all URLs and navigation
   - Event detail page route: `/events/{eventCode}`
   - Session detail route: `/events/{eventCode}/sessions/{sessionSlug}`
   - Registration route: `/events/{eventCode}/register`
   - Breadcrumbs display event code (e.g., "BATbern 56")

### Companies (AC 4-7)

4. **AC4:** Company name field enforces ID-compliant format in frontend
   - Maximum 12 characters
   - Alphanumeric only (a-z, A-Z, 0-9)
   - Real-time validation with clear error messages
   - Auto-sanitization removes invalid characters

5. **AC5:** Company entity has both `name` (ID) and `displayName` (UI label)
   - `name` used in URLs and as identifier (e.g., "GoogleZH")
   - `displayName` shown in UI (e.g., "Google Zürich AG")
   - Both fields required when creating company
   - Backend validates `name` format server-side

6. **AC6:** API endpoints accept company name as path parameter
   - `GET /api/v1/companies/{companyName}` returns company (case-insensitive)
   - `PUT /api/v1/companies/{companyName}` updates company
   - Company name lookup is case-insensitive ("GoogleZH" = "googlezh")

7. **AC7:** Company name validation provides user-friendly feedback
   - Frontend shows character count (e.g., "8/12 characters")
   - Error message: "Only letters and numbers allowed (no spaces)"
   - Suggestion: "Example: GoogleZH, UBS, Swisscom"

### Users (AC 8-11)

8. **AC8:** Username auto-generated when creating new user
   - Format: `{firstname}.{lastname}` (lowercase, slugified)
   - German characters converted (ä→ae, ö→oe, ü→ue)
   - Non-letter characters removed
   - Example: "Peter Müller" → "peter.mueller"

9. **AC9:** Username is editable in frontend during create and update
   - Username field shown with auto-generated value pre-filled
   - User can edit to custom username (must follow format)
   - Refresh button regenerates from first/last name
   - Real-time validation shows format requirements

10. **AC10:** Collision handling appends numeric suffix automatically
    - If "john.doe" exists, next is "john.doe.2"
    - Then "john.doe.3", "john.doe.4", etc.
    - Backend handles collision detection via unique constraint
    - Frontend displays collision error, offers incremented version

11. **AC11:** Speaker/Organizer/Partner routes use same username (role-based views)
    - `/users/john.doe` - Generic user profile
    - `/speakers/john.doe` - Speaker-specific view (same user)
    - `/organizers/john.doe` - Organizer-specific view (same user)
    - No separate "SpeakerId" - roles are properties on User entity

### Sessions (AC 12)

12. **AC12:** Session slug auto-generated from title, unique within event
    - Slugify session title (lowercase, hyphens, remove special chars)
    - Handle German characters (ä→ae, ö→oe, ü→ue, ß→ss)
    - Collision handling within event: append `-2`, `-3`, etc.
    - URL format: `/events/{eventCode}/sessions/{sessionSlug}`
    - Example: "Blockchain Security 101" → "blockchain-security-101"

### Technical Design Notes

**Slug Generation Algorithm:**

```java
private String slugify(String text) {
    return text.toLowerCase()
        // German characters
        .replaceAll("ä", "ae")
        .replaceAll("ö", "oe")
        .replaceAll("ü", "ue")
        .replaceAll("ß", "ss")
        // French characters
        .replaceAll("é", "e")
        .replaceAll("è", "e")
        .replaceAll("à", "a")
        .replaceAll("ê", "e")
        .replaceAll("ç", "c")
        // Remove diacritics
        .normalize("NFD")
        .replaceAll("\\p{M}", "")
        // Convert to URL-safe
        .replaceAll("[^a-z0-9]+", "-")
        .replaceAll("^-|-$", "")
        .substring(0, Math.min(text.length(), 100)); // Max 100 chars
}
```

**Collision Handling Pattern:**

```java
private String ensureUniqueUsername(String baseUsername) {
    if (!userRepository.existsByUsername(baseUsername)) {
        return baseUsername;
    }

    // Find next available suffix
    int suffix = 2;
    while (userRepository.existsByUsername(baseUsername + "." + suffix)) {
        suffix++;
    }
    return baseUsername + "." + suffix;
}
```

**ID Resolution Strategy:**

```java
// In controllers - accept both UUID and friendly ID
@GetMapping("/events/{idOrCode}")
public ResponseEntity<EventResponse> getEvent(@PathVariable String idOrCode) {
    Event event;

    if (isUUID(idOrCode)) {
        event = eventService.findById(UUID.fromString(idOrCode));
    } else {
        event = eventService.findByEventCode(idOrCode);
    }

    return ResponseEntity.ok(toResponse(event));
}

private boolean isUUID(String str) {
    try {
        UUID.fromString(str);
        return true;
    } catch (IllegalArgumentException e) {
        return false;
    }
}
```
