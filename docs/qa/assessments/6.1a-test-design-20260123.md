# Test Design: Story 6.1a - Magic Link Infrastructure

Date: 2026-01-23
Designer: Quinn (Test Architect)
Story: 6.1a-magic-link-infrastructure.md

## Test Strategy Overview

- **Total test scenarios**: 31
- **Unit tests**: 15 (48%)
- **Integration tests**: 13 (42%)
- **E2E tests**: 3 (10%)
- **Priority distribution**: P0: 12, P1: 11, P2: 6, P3: 2

### Test Level Rationale

This story is **infrastructure/backend only** with no frontend components. The test pyramid should be heavily weighted toward unit and integration tests:

- **Unit tests** for pure logic: token generation, hash computation, expiry validation
- **Integration tests** for database operations and service layer with real PostgreSQL
- **E2E tests** only for critical API contract validation via Bruno

## Recommended Execution Order

### Phase 1: Unit Tests (Fail Fast)
1. Token generation tests (UNIT-001 through -004)
2. Token validation logic (UNIT-005 through -008)
3. Security tests (UNIT-009, -010)

### Phase 2: Integration Tests (Database)
1. Repository tests (INT-001 through -004)
2. Validation flow (INT-005 through -009)
3. Single-use enforcement (INT-010 through -014)
4. API endpoint tests (INT-016 through -021)

### Phase 3: E2E Tests (API Contract)
1. Happy path validation (E2E-001)
2. Error response formats (E2E-002)
3. HTTPS enforcement (E2E-003) - staging only
