## Critical Risks Requiring Immediate Attention

### 1. SEC-002: Token Theft via Logging

**Score: 9 (Critical)**

**Probability**: High - Developers commonly log request parameters for debugging. Without explicit safeguards, tokens WILL appear in logs.

**Impact**: High - Exposed tokens allow attackers to impersonate speakers, accept/decline invitations, access portals. Potential GDPR violation if speaker data is accessed.

**Affected Components**:
- `SpeakerPortalTokenController`
- `MagicLinkService`
- API Gateway access logs
- Application logs

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Create `@MaskedToken` annotation that prevents logging | dev | Before PR |
| Preventive | Add logback filter to mask token patterns | dev | Before PR |
| Preventive | Never log request body in controller | dev | Before PR |
| Detective | Add log audit check in CI pipeline | dev | Before merge |
| Corrective | Document log retention policy for security incidents | ops | Before deploy |

**Testing Requirements**:
- Test 6.1: `should_neverLogPlaintextToken_when_validating`
- Manual log review during code review
- CI check for `token` string in log statements

**Residual Risk**: Low - With masking in place, accidental exposure unlikely

---

### 2. OPS-001: No Monitoring for Failed Validations

**Score: 9 (Critical)**

**Probability**: High - Without explicit implementation, no alerting will exist for brute force attempts or token enumeration.

**Impact**: High - Attack attempts go unnoticed, no audit trail for security incidents, compliance gaps.

**Affected Components**:
- `MagicLinkService.validateToken()`
- CloudWatch metrics/logs
- Alerting infrastructure

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Log failed attempts with IP, timestamp, token hash prefix | dev | Task 7 |
| Detective | Create CloudWatch metric for failed validations | dev | Task 7 |
| Detective | Set up alert for >10 failures/minute from same IP | ops | Before deploy |
| Corrective | Document incident response for token attacks | ops | Before deploy |

**Testing Requirements**:
- Test 6.3: `should_logFailedAttemptWithIp_when_validationFails`
- Integration test verifying audit records created

**Residual Risk**: Medium - Sophisticated attackers may use distributed IPs

## High Risks

### 3. SEC-001: Token Prediction/Enumeration Attack

**Score: 6 (High)**

**Probability**: Medium - Requires attacker to understand token format
**Impact**: High - Could access any speaker's portal

**Mitigation**:
- Use 32 bytes of SecureRandom (256 bits of entropy)
- Rate limit validation endpoint (5 req/min/IP)
- No error differentiation (always return generic 401)

**Testing**: Test 6.1: `should_generateCryptographicallySecureToken_when_called`

---

### 4. SEC-003: Rate Limiting Bypass

**Score: 6 (High)**

**Probability**: Medium - Distributed attacks could bypass IP-based limiting
**Impact**: High - Enables brute force attempts

**Mitigation**:
- Implement bucket4j or Spring rate limiter
- Consider token-based rate limiting (per token hash prefix)
- Add WAF rules at CloudFront level

**Testing**: Test 5.5: `should_return429_when_rateLimitExceeded`

---

### 5. SEC-005: Token Replay After Single-Use

**Score: 6 (High)**

**Probability**: Medium - Race condition window between validation and marking used
**Impact**: High - RESPOND tokens could be used multiple times

**Mitigation**:
- Use SELECT FOR UPDATE or database-level constraint
- Mark token used in same transaction as action
- Consider optimistic locking with version column

**Testing**: Test 3.2: `should_rejectSecondUse_when_respondTokenAlreadyUsed`

---

### 6. TECH-001: Migration Breaks Existing speaker_pool

**Score: 6 (High)**

**Probability**: Medium - Complex migration with state mappings
**Impact**: High - Breaks existing speaker data, rollback difficult

**Mitigation**:
- Test migration against production data snapshot
- Create rollback migration (V44_1__Rollback)
- Backup speaker_pool before deployment

**Testing**: Migration tests with Testcontainers

---

## Medium Risks

### 7. PERF-001: Token Validation Bottleneck

**Score: 4 (Medium)**

**Mitigation**:
- Index on `token_hash` (specified in migration)
- Connection pooling configured
- Consider caching valid tokens (short TTL)

---

### 8. DATA-001: Token Expiry Timezone Issues

**Score: 4 (Medium)**

**Mitigation**:
- Use TIMESTAMP WITH TIME ZONE in database
- Always use Instant (UTC) in Java
- Test with servers in different timezones

---

## Low Risks

### 9. SEC-004: Timing Attack on Validation

**Score: 2 (Low)**

**Mitigation**: Use constant-time comparison for hash lookup

---

### 10. TECH-002: Hash Collision (SHA-256)

**Score: 3 (Low)**

**Mitigation**: UNIQUE constraint on token_hash column

---

### 11. TECH-003: SecureRandom Seeding

**Score: 3 (Low)**

**Mitigation**: Use default SecureRandom (JVM handles seeding)
