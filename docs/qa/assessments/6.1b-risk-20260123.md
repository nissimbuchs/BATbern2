## Critical Risks Requiring Immediate Attention

### 1. SEC-001: PII Exposure in Email Logs

**Score: 9 (Critical)**

**Probability**: High - Email content with speaker names, event details, and magic links is commonly logged for debugging.

**Impact**: High - Exposure of speaker email addresses, names, and magic link tokens. GDPR violation potential. Magic links could be stolen from logs.

**Affected Components**:
- `SpeakerInvitationService`
- `EmailService`
- AWS SES request/response logging
- Application logs

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Never log email body or recipient in production | dev | Before PR |
| Preventive | Mask email addresses in logs (j***@example.com) | dev | Before PR |
| Preventive | Use structured logging with PII markers | dev | Before PR |
| Detective | Log audit for email-related entries | dev | Before merge |
| Corrective | Document data retention for email logs | ops | Before deploy |

**Testing Requirements**:
- Verify no PII in application logs during invitation
- Check AWS SES CloudWatch logs configuration

**Residual Risk**: Medium - CloudWatch logs may still contain metadata

## High Risks

### 2. TECH-001: Company Service Unavailable During Invite

**Score: 6 (High)**

**Probability**: Medium - Microservice communication can fail due to network issues, deployments, or service overload.

**Impact**: High - Cannot create user, invitation fails entirely. No graceful degradation.

**Affected Components**:
- `UserAutoCreationService`
- `UserApiClient` (HTTP client)
- Circuit breaker configuration

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Implement circuit breaker (Resilience4j) | dev | Task 2 |
| Preventive | Add retry with exponential backoff | dev | Task 2 |
| Detective | Health check for Company Service before batch | dev | Task 4 |
| Corrective | Return clear error message to organizer | dev | Task 7 |

**Testing Requirements**:
- Test 2.1 variant: `should_returnError_when_companyServiceUnavailable`
- Integration test with WireMock simulating service failure

---

### 3. TECH-002: Magic Link Service Dependency Failure

**Score: 6 (High)**

**Probability**: Medium - Story 6.1a must be implemented first. Integration issues possible.

**Impact**: High - Cannot generate tokens, invitation email has broken links.

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|

**Testing Requirements**:

---

### 4. TECH-004: Transaction Boundary Issues

**Score: 6 (High)**

**Probability**: Medium - Multiple operations (create user, create SpeakerPool, send email) span services.

**Impact**: High - User created but SpeakerPool not created, or SpeakerPool created but email not sent.

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Use saga pattern: create entities first, email last | dev | Task 4 |
| Preventive | Mark SpeakerPool as PENDING until email confirmed | dev | Task 4 |
| Detective | Scheduled job to find orphaned IDENTIFIED entries | dev | Future |
| Corrective | Manual resend option for failed emails | dev | Task 7 |

**Testing Requirements**:
- Test simulating email failure after SpeakerPool creation
- Verify rollback or compensation logic

---

### 5. OPS-001: AWS SES Sending Limits Exceeded

**Score: 6 (High)**

**Probability**: Medium - SES has sending quotas (sandbox: 200/day, production: varies).

**Impact**: High - Batch invitations fail mid-way, some speakers not notified.

**Mitigation**:
| Strategy | Action | Owner | Timeline |
|----------|--------|-------|----------|
| Preventive | Check remaining quota before batch | dev | Task 5 |
| Preventive | Implement rate limiting for batch (e.g., 10/sec) | dev | Task 5 |
| Detective | Alert when approaching 80% of daily quota | ops | Before deploy |
| Corrective | Queue emails for later delivery if quota exceeded | dev | Future |

**Testing Requirements**:
- Test 5.3: `should_continueProcessing_when_oneInvitationFails`
- Verify partial success handling

## Medium Risks

### 6. DATA-001: Duplicate SpeakerPool Entries

**Score: 4 (Medium)**

**Mitigation**:
- Unique constraint on (event_id, username)
- Check existence before insert (idempotency)
- Test 7.3: `should_returnExistingSpeakerPool_when_duplicateInvite`

---

### 7. OPS-002: Email Bounces Not Tracked

**Score: 4 (Medium)**

**Mitigation**:
- Configure SES bounce/complaint notifications
- Store delivery status in SpeakerPool or separate table
- Future: webhook for bounce handling

---

### 8. OPS-003: No Retry Mechanism for Failed Emails

**Score: 4 (Medium)**

**Mitigation**:
- Implement @Retryable on email sending
- Store failed attempts for manual retry
- Organizer UI shows "resend" button

---

### 9. PERF-001: Batch Invitation Blocks for Large Batches

**Score: 4 (Medium)**

**Mitigation**:
- Process batch asynchronously (@Async)
- Return immediately with job ID
- Polling endpoint for batch status

---

### 10. BUS-001: Email Goes to Spam Folder

**Score: 4 (Medium)**

**Mitigation**:
- Configure SPF, DKIM, DMARC for batbern.ch
- Use verified sender identity in SES
- Test with mail-tester.com
