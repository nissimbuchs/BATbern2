# Story 10.24: AWS Cognito User Provisioning from User Management

Status: ready-for-dev

## Story

As an **organizer**,
I want to create an AWS Cognito account for a BATbern user directly from the User Management list — either when creating a new user or for an existing user without a Cognito account —
so that I can onboard people without requiring them to self-register.

## Acceptance Criteria

1. `POST /api/v1/users/{username}/provision-cognito` creates a Cognito user with `FORCE_CHANGE_PASSWORD` state; links `cognitoUserId` in DB; sends custom welcome email; organizer-only (403 for non-organizers)
2. Backend uses ECS IAM task role for `AdminCreateUser` — the organizer's own Cognito token has no AWS IAM rights; confirmed by `CognitoAdminServiceTest` mocking the SDK client (not Cognito tokens)
3. 409 returned if email already exists in Cognito (`UsernameExistsException`) — no duplicate created, DB unchanged
4. 409 returned if user already has a `cognitoUserId` in DB (already provisioned guard)
5. `UserResponse` includes `hasCognitoAccount: boolean` (does NOT expose raw `cognitoUserId`)
6. User table row action menu shows "Create AWS Account" only for users with `hasCognitoAccount = false`
7. `LinkCognitoDialog` shows confirmation, language selector, success/error states correctly
8. Create-user modal toggle "Also create AWS account" triggers provisioning after DB user is created; when Cognito provisioning fails after user creation: (a) BATbern user record is still persisted, (b) `cognitoUserId` remains null, (c) warning toast shown "User created but AWS account creation failed — use 'Create AWS Account' from the user list to retry", (d) integration test asserts DB state and toast presence
9. IAM: CDK grants `AdminCreateUser` + `AdminAddUserToGroup` scoped to the specific user pool ARN
10. Welcome email templates seeded and visible/editable in Email Templates admin tab
11. TDD: `CognitoAdminServiceTest` covers success, `UsernameExistsException`, SDK error, already-linked guard
12. OpenAPI spec (`users-api.openapi.yml`) committed before implementation (ADR-006)
13. i18n: `actions.createAwsAccount`, `linkCognito.*` keys in de/en `userManagement` namespace; Type-check passes; Checkstyle passes

## Dev Notes

### Path Variable Clarification — `{username}` NOT `{userId}`

The story spec says `POST /api/v1/users/{userId}/provision-cognito`, but the **existing** `UserController` uniformly uses `{username}` (the meaningful public identifier per Story 1.16.2 / ADR-003). The endpoint must be:
```
POST /api/v1/users/{username}/provision-cognito
```
In the frontend, `user.id` **is** the username (UserResponse.id = username — see OpenAPI `UserResponse.id` description: "Username (unique identifier)"). So `apiClient.post('/api/v1/users/' + user.id + '/provision-cognito', ...)` is correct.

### Why Cognito SDK Calls Work from CUMS ECS Task Role

The organizer's browser sends a JWT to the API Gateway. CUMS validates the JWT for authorization (ORGANIZER role). For the actual Cognito `AdminCreateUser` call, CUMS's ECS task runs with an **IAM task role** that has `cognito-idp:AdminCreateUser` permission. This is completely separate from the user's JWT token. The AWS SDK in Java picks up IAM credentials automatically via the EC2 instance metadata / container credentials endpoint — no manual credential setup needed.

### Cognito SDK — `AdminCreateUser` Pattern

```java
    .userPoolId(userPoolId)
    .username(user.getEmail())          // email as username
```

### GlobalExceptionHandler — Adding 409 for CognitoProvisioningConflictException

Follow the existing `CompanyValidationException` pattern (line 116-131 in `GlobalExceptionHandler.java`):
```java
@ExceptionHandler(CognitoProvisioningConflictException.class)
public ResponseEntity<ErrorResponse> handleCognitoProvisioningConflictException(
        CognitoProvisioningConflictException ex,
        HttpServletRequest request) {
    log.warn("Cognito provisioning conflict: {}", ex.getMessage());
    ErrorResponse error = ErrorResponse.builder()
            .timestamp(Instant.now())
            .path(request.getRequestURI())
            .status(HttpStatus.CONFLICT.value())
            .error("Conflict")
            .message(ex.getMessage())
            .correlationId(CorrelationIdGenerator.generate())
            .severity("WARNING")
            .build();
    return ResponseEntity.status(HttpStatus.CONFLICT).body(error);
}
```

```java
    });

Map<String, String> vars = Map.of(
    "recipientName", user.getFirstName() + " " + user.getLastName(),
    "loginUrl", baseUrl + "/login",
    "temporaryPasswordNote", language.equals("de")
        ? "Bitte ändern Sie Ihr Passwort beim ersten Login."
        : "Please change your password on first login."
);
String body = emailService.replaceVariables(html, vars);
// subject is stored in DB template; parse it from htmlBody or keep in a separate field
emailService.sendHtmlEmail(user.getEmail(), subject, body);
```

### Testing Pattern — Mocking `CognitoIdentityProviderClient`

Use `@MockitoBean` (Spring Boot 3.4+) if the project is already on that version; otherwise `@MockBean`:
```java
@MockitoBean
private CognitoIdentityProviderClient cognitoClient;
```
Mock `adminCreateUser()` and `adminAddUserToGroup()` to return success responses; mock `UsernameExistsException` for conflict tests.

Do NOT use `when(cognitoClient.adminCreateUser(...)).thenReturn(...)` — use the builder pattern:
```java
// Success mock
AdminCreateUserResponse mockResponse = AdminCreateUserResponse.builder()
    .user(UserType.builder().username("test@example.com").build())
    .build();
when(cognitoClient.adminCreateUser(any(AdminCreateUserRequest.class)))
    .thenReturn(mockResponse);

// UsernameExistsException mock
when(cognitoClient.adminCreateUser(any(AdminCreateUserRequest.class)))
    .thenThrow(UsernameExistsException.builder().message("User already exists").build());
```
