## Challenges & Growth Areas ⚠️

### 1. Recurring API Contract Surprises
Entity/table naming mismatches discovered during implementation, not during planning:
- W2.1: FK target is `user_profiles` not `users`; OpenAPI spec lives in `users-api.openapi.yml` not `company-user-management-api.openapi.yml`
- W2.2: `AuthResponse.expiresAt` used `LocalDateTime` (no timezone) → Swift silent fallback (HIGH bug)
- W2.3: Event entity doesn't have `typicalStartTime`/`typicalEndTime` — derived from sessions in controller; `PortraitCache` existing API differed from story spec

**Root cause:** Watch stories specify API shapes based on the architecture doc, but the backend domain model evolves. No sync between arch doc and actual implementation.

### 2. Security Bugs in Two Stories
- W2.1: `ThreadLocalRandom` used for pairing codes → replaced with `SecureRandom` in code review
- W2.2: TOCTOU race in pairing code validation → fixed with `@Transactional claimPairingCode()`

Both caught in code review — not pre-implementation review. These are the kinds of bugs that should be in a security checklist before implementation.

### 3. N+1 Queries in Two Stories
- W2.3: Speaker loading in `mapToSessionDetail()` — one `findByUsername()` per speaker
- W2.4: `getTotalSpeakerCount()` — N queries fixed with `countDistinctSpeakersByEventCode`

Pattern: when mapping speaker lists, always use batch queries. This should be in story templates for any endpoint that returns speaker collections.

### 4. Domain Assumption Error (W2.2 — HIGH)
`findActiveEventsForOrganizer` was filtering by `organizerUsername` — but all organizers manage all events on the Watch. Caught in code review, required changing the JPQL query, test, and controller. Impact: if shipped, each organizer would only see their assigned event.

### 5. Technical Debt Deferred Without Tracking
W2.3 deferred `companyLogoUrl` cross-service lookup and `SpeakerRole` enum mapping with "documented in completion notes" — but no tracking entry was created at the time. These were discovered retroactively during the retrospective.

---

## Key Insights 💡

1. **Pre-implementation adversarial review is mandatory from W3.x onwards.** W2.4 proved it catches more issues at lower cost than post-implementation review.

2. **Security checklist needed.** Two stories had security bugs caught only in review. A per-story security checklist (SecureRandom, @Transactional for TOCTOU-sensitive ops, input validation) would surface these earlier.

3. **N+1 is a recurring pattern for speaker endpoints.** Add "use batch queries for speaker collections" to the story template for any Watch API endpoint that returns speaker data.

4. **Domain assumptions must be validated against the actual entity model** before writing a story, not during implementation. Architecture doc ≠ actual domain model.

5. **Deferred debt needs a tracking entry at deferral time.** "Documented in completion notes" is not enough — it won't surface unless someone reads the full story file.
