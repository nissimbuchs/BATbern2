## What Went Well ✅

### 1. Real-World Multi-Watch Validation — First Run
Nissim tested on real hardware with multiple watches. The Epic 2 WebSocket sync (arrival tracking), Live Countdown, and Complications all worked on the first run. The wall-clock timer design — where all watches independently calculate from the same `scheduledEndTime` without network coordination — proved out in practice. This is the highest-quality validation we can get.

### 2. Complications Worked End-to-End
The W3.3 Widget Extension target setup (manual Xcode steps, App Group capability, multi-target file membership, `WKCompanionAppBundleIdentifier`) was complex. The instructions landed well enough that Nissim could complete the setup and get complications appearing on the Watch face. WidgetKit + App Groups first-run success is not guaranteed.

### 3. W3.4 Pragmatic Simplification
The team recognised mid-story that a dedicated O7 screen was unnecessary — the existing `SessionListView` (Epic 1) covered 80% of the requirement. The decision to add status badges to the existing view rather than build greenfield saved significant scope. The right kind of simplification.

### 4. Pre-Implementation UX Review on W3.3
W3.3 had an explicit pre-implementation design alignment review that caught three issues before a line was written: C3 corner showing MM:SS (too wide for `.accessoryCorner`), wrong urgency green color (`#22c55e` not `.teal`), and missing overtime display states. All fixed before implementation.

### 5. Protocol-Based DI Carried Over Cleanly
`WatchHapticService`, `ComplicationDataStore`, `SessionBadgeStatus` — all new components shipped with clear boundaries and testable interfaces. Fast, deterministic tests throughout.

## Challenges & Growth Areas ⚠️

### 1. W3.1 Wrong Design Direction Implemented
The dev implemented a full-screen progress ring (Direction 3 style) when the chosen design was explicitly "Progress Ring + Card Stack" with a compact 96×96pt ring. Required a full rebuild of `LiveCountdownView` post-review. Root cause: the chosen design direction was documented in Task 2.4 subtasks, not prominently flagged at story top level.

### 2. W3.2 Required Two Review Cycles
The first review cycle missed that `DispatchQueue.main.asyncAfter` for multi-tap haptic patterns was non-cancellable — orphaned haptics could fire after `stopEventSession()`. Caught in cycle 2, replaced with cancellable `Task`. Also: `WKExtendedRuntimeSessionDelegate` inferring `@MainActor` conflicting with `WatchHapticService()` default parameter expression was a WatchKit subtlety not anticipated pre-implementation.

**Root cause:** No formal pre-implementation adversarial review for W3.2. The WatchKit threading complexity was not surfaced until implementation was complete.

### 3. Pre-Implementation Review Still Not Landing Consistently
Epic 2 retro action item 1 — "add pre-implementation adversarial review to every story" — was reported as ⏭️ "applied inline per story." In practice: W3.3 had it explicitly. W3.1, W3.2, W3.4 did not. W3.2 paid the price with two review cycles.

### 4. Architectural Overlap — Public Zone vs Organizer Zone
**The most significant finding of this retrospective (raised by Nissim):**

W3.4's simplification revealed a deeper architectural gap: the `SessionListView` / `SessionCardView` designed for Epic 1 (public/attendee zone) already covered a large portion of what W3.4 was meant to add for organisers. The architect did not audit what Epic 1 had built when designing Epic 3 stories, leading to:
- EventController refactoring discovered mid-implementation
- Offline status handled separately in public and organizer zones when both should use the same `EventStateManager` state
- Duplication risk going forward

**Nissim's concern (verbatim):** "a better redesign upfront would have helped. I am afraid that in Epic 4 we will encounter similar problems — things already implemented in public view and the danger of reimplementing them in Epic 4 for organizer views."

## Key Insights 💡

1. **First-run real-hardware success is the product of good architectural decisions.** Wall-clock timer, shared `EventStateManager`, App Group data store — these design choices from Epic 1-2 paid compounding returns in Epic 3.

2. **The pre-implementation adversarial review is still not consistent.** W3.2 proves the cost of skipping it. For Epic 4 — WebSocket state machine + conflict resolution — the complexity is higher, not lower. This must be a documented, non-skippable story step.

3. **Design direction must be pinned at story top level.** Not buried in task subtasks. W3.1's rebuild was avoidable.

4. **The public zone and organizer zone share more than the architecture doc implied.** Before any new organizer-zone screen is designed, check: does a public-zone component already do 70% of this?

5. **Epic 4 has a structural risk of reimplementation.** O6 (W4.2 transition view) overlaps with `LiveCountdownView.nextSessionCard()`. O5 (W4.4 break view) overlaps with `SessionCardView` break cards and `LiveCountdownViewModel` break gong routing. W4.1 WebSocket connectivity state risks fragmenting from existing `ConnectionStatusBar`.

## Action Items 📝

### Process Improvements

**A1 — Pre-implementation adversarial review: mandatory, documented**
Owner: Bob (SM)
What: Add as an explicit, non-skippable written section in every story file before the Dev agent starts. The W3.3 format (UX alignment notes captured before Task 1) is the template. Not "inline" — it must appear as a committed section in the story file.
Success criteria: Every W4.x story has a completed "Pre-Implementation Review" section before Dev agent is assigned.

**A2 — Design direction pinning at story top level**
Owner: Bob (SM)
What: When a story references a UX design with multiple options, the chosen direction must be called out in a dedicated `## Design Direction (CHOSEN)` block at the top of the story — not buried in task subtasks.
Success criteria: All W4.x stories with UI components have a prominent chosen-direction block visible before any task description.

### Technical Debt

**D1 — WatchHapticService `schedule()` stub**
Owner: Amelia (Dev)
Priority: High
What: `schedule(_ alert:, at:)` currently appends to `scheduledQueue` but never fires. W4.1 server-triggered haptic scheduling requires this to fire at the scheduled time via an Extended Runtime session timer loop.
Success criteria: Implemented, tested, and verified before W4.1 starts.

### Team Agreements

**T1 — Public zone first, organizer zone second**
Before designing any new organizer-zone screen, ask: *does a public-zone component already do 70% of this?* If yes, extend — don't duplicate. This applies to all future epics.
