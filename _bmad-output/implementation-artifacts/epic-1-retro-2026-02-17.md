## What Went Well ‚úÖ

### 1. Test Framework Scaffold (TEA) Paid Off Massively
The `testarch-framework` workflow scaffolded test infrastructure before W1.1 began. Pre-existing mocks (`MockAPIClient`, `MockClock`, `MockWebSocketClient`, `MockHapticService`), test factories, and domain logic tests (SessionTimerEngine, HapticScheduler) meant every story started with a solid testing foundation. This is a practice to repeat for every new project area.

### 2. Working App on Apple Watch After Story 1
The vertical slice approach delivered a running Apple Watch app with real event data after W1.1 alone. Not a prototype ‚Äî a real, deployable app. This validated the story sequencing and gave immediate confidence.

### 3. CLI Xcode Build Enabled Full Automation
`xcodebuild` CLI support meant no GUI dependency for building and testing. The entire Watch app pipeline is scriptable, enabling automation and consistent CI behaviour.

### 4. Protocol-Based Dependency Injection from Day 1
MVVM + protocol contracts (`APIClientProtocol`, `ClockProtocol`, `HapticServiceProtocol`) made every component testable in isolation. Fast, deterministic tests with no network dependency throughout all 4 stories.

### 5. Progressive Publishing Handled Correctly
The TOPIC ‚Üí SPEAKERS ‚Üí AGENDA phase logic was implemented incrementally across W1.2, W1.3, W1.4 and works correctly. The Watch respects publishing phases ‚Äî a critical business rule delivered cleanly.

## Challenges & Growth Areas ‚öÝÔ∏è

### 1. Localization / Swiss German Not First-Class from Day 1
**What happened:** German language (`de_CH`) and Swiss German formatting (`SwissDateFormatter`, `RelativeDateTimeFormatter`) were added reactively ‚Äî story by story ‚Äî rather than being designed as first-class constraints from W1.1.

**Impact:** Rework in each story adding `NSLocalizedString()` calls and date formatters after views were already built.

**Root cause:** The Watch app was designed as if it were a generic app, not as a Swiss-context app from the start.

### 2. Missed Platform Pattern Reuse
**What happened:** The BATbern web frontend and backend already had proven patterns ‚Äî API call shapes, i18n keys, loading state patterns, error handling ‚Äî that weren't consulted until mid-epic.

**Impact:** Reinvented patterns that already existed. The `SessionSpeakerResponse` missing `bio` field was discovered mid-W1.3 rather than during story planning.

**Root cause:** Watch app treated as greenfield instead of brownfield extension of the existing platform.

### 3. Reactive State Management Pattern Not Documented Clearly
**What happened:** W1.4 code review found 10 issues including polling instead of callbacks, missing data comparison logic, and incomplete tests in the connectivity monitoring implementation.

**Impact:** Rework during W1.4 code review phase.

**Root cause:** Callback-based reactive patterns for `NWPathMonitor` weren't documented or referenced before implementation.

### 4. Progressive Publishing Logic Spread Across 3 Stories
**What happened:** The same files (`SessionCardView`, `PublicViewModel`) were revisited in W1.2, W1.3, and W1.4 to incrementally harden progressive publishing.

**Impact:** Multiple returns to the same codebase areas ‚Äî not ideal for clean story boundaries.

**Root cause:** The full publishing logic wasn't fully designed upfront; it evolved story-by-story.

## Key Insights üí°

1. **Watch app is brownfield, not greenfield.** The BATbern platform is a rich reference ‚Äî web frontend components, API contracts, i18n keys, and Spring Boot service patterns should all be consulted before writing any Watch story.

2. **Localization is a first-class design constraint for a Swiss product.** German language and Swiss formatting must be specified in story requirements, not discovered during implementation.

3. **Pre-story platform scan is mandatory.** Before writing any story, ask: "Does the web frontend or an existing service already do something similar?" Then reference that pattern.
